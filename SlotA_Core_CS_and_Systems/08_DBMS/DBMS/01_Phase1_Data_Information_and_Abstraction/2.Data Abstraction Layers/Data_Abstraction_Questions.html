SECTION 1 — DATABASE FUNDAMENTALS

1. What is a database?
A database is an organized collection of data that is stored electronically. Think of it as a digital filing cabinet where information is stored in a structured way so it can be easily accessed, managed, and updated.

2. What is a DBMS?
DBMS stands for Database Management System. It’s software that helps you create, manage, and use databases. Examples include MySQL, Oracle, and PostgreSQL. It’s like a librarian that organizes and retrieves your data.

3. Why do we need DBMS?
We need DBMS because manually managing large amounts of data is difficult and error-prone. DBMS helps store data efficiently, retrieve it quickly, keep it secure, and ensure it remains accurate and consistent.

4. What problems exist in file systems?
File systems (like storing data in Excel or text files) have problems like:

  • Data redundancy (same data stored multiple times)
  • Data inconsistency (updates don’t reflect everywhere)
  • No easy way to search or query data
  • Data isolation (data scattered in different files)
  • No security controls
  • No backup/recovery mechanisms

5. How does DBMS manage data?
DBMS manages data by:

  • Storing it in structured tables
  • Using indexes for fast searching
  • Enforcing rules (constraints) to keep data valid
  • Managing who can access what data
  • Handling backups and recovery
  • Optimizing queries for speed

6. What are responsibilities of DBMS?

  • Data storage and retrieval
  • Security and access control
  • Data integrity (accuracy)
  • Concurrency control (multiple users at once)
  • Backup and recovery
  • Query optimization

7. What are examples of DBMS?

  • MySQL (popular, free)
  • PostgreSQL (advanced, free)
  • Oracle (enterprise)
  • Microsoft SQL Server
  • MongoDB (NoSQL)
  • SQLite (lightweight)

8. What is relational DBMS?
A Relational DBMS (RDBMS) stores data in tables with rows and columns, where tables can be related to each other. It uses SQL to manage data. Examples: MySQL, Oracle, PostgreSQL.

9. What is SQL?
SQL (Structured Query Language) is the standard language used to communicate with databases. You use it to insert, update, delete, and retrieve data.

10. Why is SQL important?
SQL is important because:

  • It’s the universal language for databases
  • It’s easy to learn
  • It allows complex queries
  • It’s standardized across different database systems
  • It’s essential for data analysis and backend development

SECTION 2 — DATA ABSTRACTION

11. What is data abstraction?
Data abstraction means hiding complex details and showing only essential information. Like driving a car—you use the steering wheel and pedals without knowing how the engine works internally.

12. Why is data abstraction needed?
Because databases are complex. Abstraction lets users interact with data simply without needing to understand all the technical details of how data is stored or processed.

13. What complexity does DBMS hide?

  • How data is physically stored on disk
  • How indexes work
  • How queries are optimized
  • How memory is managed
  • How concurrent access is handled

14. How does abstraction simplify databases?
It provides simple interfaces (like SQL) so users can say “give me all customers” without knowing where or how customer data is stored.

15. What are advantages of abstraction?

  • Easier to use
  • More secure (users can’t accidentally break things)
  • Easier to maintain
  • Changes can be made without affecting users
  • Reduces errors

16. What problems happen without abstraction?

  • Users must know complex technical details
  • Easy to make mistakes
  • Hard to change the system
  • Security risks
  • Very difficult to use for non-technical people

17. How does abstraction improve security?
By limiting what users can see and do. Users interact through controlled interfaces rather than directly accessing raw data.

18. How does abstraction improve maintainability?
You can change the internal workings (like switching storage methods) without changing how users interact with the database.

19. How does abstraction improve flexibility?
Different users can have different views of the same data. The system can evolve without breaking existing applications.

20. How does abstraction improve usability?
Users see only what they need in a simple format. A manager sees sales reports; a clerk sees customer forms—both using the same database but different simplified views.

21. What is the real-world meaning of abstraction?
Think of a TV remote: you press “power” and the TV turns on. You don’t need to know about electricity, circuits, or signals. That’s abstraction.

22. How is abstraction used in DBMS?
Through three layers: External (user views), Conceptual (logical structure), and Internal (physical storage). Users only see the external layer.

23. What is separation of concerns?
Dividing a system into parts where each part handles one specific thing. In DBMS, storage is handled separately from user interfaces, which is handled separately from security.

24. How does DBMS separate complexity into layers?

  • External layer: What users see
  • Conceptual layer: What the database logically contains
  • Internal layer: How data is physically stored

25. What happens internally when user executes SQL?

  1. SQL query is parsed (checked for errors)
  2. Optimized (best execution plan chosen)
  3. Translated to low-level operations
  4. Data is fetched from storage
  5. Results are formatted and returned

26. What internal operations are hidden from users?

  • Disk reads/writes
  • Index lookups
  • Memory buffering
  • Lock management
  • Query plan selection

27. How does DBMS hide storage details?
Users write SELECT * FROM Customers without knowing if data is on SSD, HDD, or distributed across servers.

28. How does DBMS hide indexing details?
Users don’t need to know that an index exists or how it speeds up searches. The DBMS automatically uses indexes when helpful.

29. How does DBMS hide query optimization?
The DBMS automatically finds the fastest way to run your query. You write a simple query; the DBMS decides whether to use an index, which join method, etc.

30. Why should users not interact with physical storage directly?
Because they could:

  • Accidentally corrupt data
  • Bypass security
  • Cause performance problems
  • Break the database structure

SECTION 3 — ANSI/SPARC ARCHITECTURE

31. What is ANSI/SPARC architecture?
It’s a standard three-level architecture for database systems proposed by ANSI/SPARC committee in 1975. It separates database systems into three layers: External, Conceptual, and Internal.

32. Why was ANSI/SPARC introduced?
To standardize database architecture, promote data independence, and ensure databases could evolve without breaking applications.

33. What problems existed before ANSI/SPARC?

  • Applications were tied to specific physical storage
  • Changing storage format broke applications
  • No standard way to design databases
  • Hard to share data between applications

34. What are the three levels of DBMS architecture?

  1. External Level: User views
  2. Conceptual Level: Logical database design
  3. Internal Level: Physical storage

35. What is the purpose of 3-level architecture?
To separate user applications from physical storage, allowing changes at one level without affecting others.

36. How does ANSI/SPARC architecture work?
Data flows from physical storage (Internal) → logical structure (Conceptual) → user views (External). Mappings between layers translate requests.

37. What is abstraction hierarchy in DBMS?
The three levels form a hierarchy from most abstract (External - what users see) to least abstract (Internal - physical bytes on disk).

38. Why is layered DBMS design important?
It provides flexibility, independence, and easier maintenance. You can change one layer without affecting others.

39. How do abstraction levels communicate?
Through mappings (translations). When a user queries a view, it’s mapped to the conceptual schema, then to the physical storage.

40. What is schema mapping?
The process of translating requests between levels. For example, converting a view name to actual table names and then to physical file locations.

41. What is external-conceptual mapping?
Translates user views (external) to the logical database structure (conceptual). A view might combine multiple tables; this mapping handles that translation.

42. What is conceptual-internal mapping?
Translates logical structure (conceptual) to physical storage (internal). It decides which files, indexes, and storage structures to use.

43. How does data move between layers?
Through mappings: External request → Conceptual translation → Internal translation → Physical access → Results flow back up.

44. What are advantages of layered architecture?

  • Data independence
  • Better security
  • Easier maintenance
  • Multiple user views
  • Simplified development

45. What are disadvantages of layered architecture?

  • Slight performance overhead (mappings take time)
  • More complex to design initially
  • Harder to debug across layers

46. How does layered architecture improve scalability?
You can scale physical storage (add more servers) without changing applications or logical design.

47. How does layered architecture improve maintainability?
Fixing or upgrading one layer (like switching storage engines) doesn’t require changes to other layers.

48. How does layered architecture improve security?
Users only access external views, never physical storage. Sensitive data can be hidden at the conceptual level.

49. How does layered architecture improve portability?
Applications work the same way even if you move from one database system to another or change hardware.

50. How does layered architecture improve optimization?
The internal layer can be optimized (new indexes, better storage) without affecting users or applications.

SECTION 4 — EXTERNAL / VIEW LEVEL

51. What is external level?
The highest level of abstraction. It’s what users and applications actually see and interact with—customized views of the database.

52. What is view level?
Same as external level. It’s the “view” that users have of the data, which may be different from how data is actually stored.

53. What is external schema?
The definition of a specific user view. It describes what data a particular user or application can see and how it’s presented.

54. What is view schema?
Same as external schema. It defines the structure of a database view.

55. What is subschema?
An old term for external schema. It’s a subset of the database schema tailored for specific users.

56. Why do we need external views?

  • Different users need different data
  • Security (hide sensitive data)
  • Simplify complex data for users
  • Present data in familiar formats

57. What is user-specific data representation?
Showing each user only the data they need in a format they understand. A manager sees summaries; an analyst sees detailed reports.

58. How do different users see different data?
Through views. The same database can show a salesperson only their customers, while a manager sees all customers.

59. How do views improve security?
By restricting what users can see. You can create a view that hides salary columns so employees see only names and departments.

60. What is view-level security?
Controlling access at the view level. Users are granted permissions on views rather than underlying tables, limiting their access.

61. What is row filtering?
Showing only certain rows in a view. Example: CREATE VIEW ActiveCustomers AS SELECT * FROM Customers WHERE Status = 'Active'

62. What is column filtering?
Showing only certain columns. Example: A view that shows employee names but hides their salaries.

63. What is customized data access?
Tailoring data presentation for specific needs. A view might combine data from multiple tables and calculate totals for a manager’s dashboard.

64. What is data hiding using views?
Concealing sensitive or irrelevant data from users by only exposing specific columns or rows through views.

65. What are examples of external views?

  • Sales report view (only sales data)
  • Employee directory (no salary info)
  • Active orders view (excludes completed orders)
  • Customer summary (aggregated data)

66. What is virtual table?
A view is called a “virtual table” because it looks and acts like a table but doesn’t actually store data—it generates data from underlying tables when queried.

67. What is a database view?
A saved query that appears as a table. It doesn’t store data itself but shows data from one or more tables based on defined criteria.

68. What is materialized view?
A view that actually stores (materializes) the query results physically. It updates periodically, trading storage space for faster query speed.

69. Difference between virtual view and materialized view?

  • Virtual view: Computed on-the-fly, always current, slower for complex queries
  • Materialized view: Pre-computed and stored, may be stale, faster to query

70. What is view abstraction?
Using views to hide the complexity of underlying table structures, joins, and calculations from users.

71. Can multiple views exist for same database?
Yes! You can create hundreds of views on the same tables, each tailored to different users or purposes.

72. What users interact with external level?
End users, application developers, report generators—anyone using the database through applications or queries.

73. How do applications use views?
Applications query views as if they were tables. The view handles the complex joins and filtering behind the scenes.

74. How does external layer simplify databases?
Users see simple, relevant data instead of complex table structures with many relationships and technical fields.

75. What is computed column in views?
A column in a view that is calculated from other columns, like TotalPrice = Quantity * UnitPrice.

76. What is aggregation in views?
Using functions like SUM, COUNT, AVG in views to show summarized data rather than raw details.

77. What is view updatability?
Whether you can insert, update, or delete data through a view. Simple views usually allow this; complex views with joins/aggregations often don’t.

78. What are limitations of views?

  • Performance overhead (especially complex views)
  • Not all views are updatable
  • May hide important data that users actually need
  • Can become outdated if underlying tables change

SECTION 5 — CONCEPTUAL / LOGICAL LEVEL

79. What is conceptual level?
The middle layer that describes the entire database logically—what tables exist, what columns they have, and how they relate, without physical storage details.

80. What is logical level?
Same as conceptual level. It’s the logical organization of all data in the database.

81. What is conceptual schema?
The blueprint of the entire database: all tables, columns, relationships, constraints, and business rules.

82. What is logical schema?
Same as conceptual schema. It defines the logical structure of the database.

83. What does conceptual level describe?

  • All entities (tables)
  • All attributes (columns)
  • All relationships between tables
  • All constraints and rules
  • The complete logical design

84. What exists at conceptual level?
Tables, columns, primary keys, foreign keys, constraints, relationships, and business rules—but not physical storage details.

85. What is logical database structure?
The organization of data into tables with defined columns, data types, and relationships, independent of how it’s stored.

86. What are entities?
Real-world objects or concepts represented as tables in the database. Example: Customer, Order, Product.

87. What are attributes?
Properties or characteristics of entities, represented as columns. Example: Customer has Name, Email, Phone attributes.

88. What are relationships?
How entities connect to each other. Example: A Customer places many Orders (one-to-many relationship).

89. What are constraints?
Rules that data must follow. Example: Email must be unique, Age must be positive, Order must have a valid Customer.

90. What are business rules?
Real-world rules that the database enforces. Example: “An employee cannot be their own manager” or “Discount cannot exceed 50%.”

91. What are domains?
The allowed set of values for an attribute. Example: Gender domain might be {Male, Female, Other}, or Salary domain is positive numbers.

92. What is relation schema?
The definition of a table: its name, columns, data types, and constraints. Example: Employee(EmployeeID INT, Name VARCHAR(100), Salary DECIMAL).

93. How are tables represented logically?
As relations with rows (tuples) and columns (attributes), following the relational model.

94. What is logical organization of data?
How data is structured conceptually—grouped into tables with relationships—rather than how it’s physically arranged on disk.

95. What is complete database blueprint?
The conceptual schema showing all tables, their columns, relationships, and constraints—the full logical design.

96. What is schema design?
The process of creating the conceptual schema: deciding what tables to create, what columns they need, and how they relate.

97. What is entity relationship modeling?
A visual technique (using ER diagrams) to design databases by identifying entities, their attributes, and relationships.

98. What is logical data modeling?
Creating a detailed logical design of the database, including tables, columns, keys, and relationships, before implementation.

99. What users work at logical level?
Database designers, architects, and administrators who design and maintain the database structure.

100. Why is conceptual level important?
It’s the foundation of the database. A good conceptual design ensures data integrity, reduces redundancy, and supports business needs.

SECTION 6 — INTERNAL / PHYSICAL LEVEL

101. What is internal level?
The lowest level of abstraction. It deals with how data is actually stored on physical storage devices.

102. What is physical level?
Same as internal level. It describes physical storage structures and access methods.

103. What is physical schema?
The definition of how data is stored physically: file organizations, indexes, compression, partitioning, etc.

104. What is storage schema?
Same as physical schema. It describes the physical storage structures.

105. What does internal level describe?

  • File structures and formats
  • Index structures
  • Data compression
  • Encryption methods
  • Storage allocation
  • Buffer management

106. What exists at physical level?
Files on disk, disk blocks, pages, indexes, buffers, and all the mechanisms for reading/writing data.

107. How is data physically stored?
As bytes in files on disk, organized into blocks or pages, often with indexes for fast access.

108. What are disk blocks?
Fixed-size chunks of disk space (typically 4KB-64KB) that the database reads/writes as a unit.

109. What are pages in DBMS?
Same as disk blocks. Pages are the basic unit of data transfer between disk and memory.

110. What is file organization?
How records are arranged in files: heap (unordered), sequential (sorted), hash-based, or index-based.

111. What are indexes?
Data structures that speed up searches by creating a map between values and their physical locations.

112. What is indexing?
The process of creating and maintaining indexes to improve query performance.

113. What is B+ Tree?
A balanced tree data structure commonly used for indexes. It keeps data sorted and allows fast search, insert, and delete operations.

114. What is hash indexing?
Using a hash function to map values directly to storage locations. Very fast for exact matches, but doesn’t support range queries.

115. What is bitmap indexing?
Using bitmaps (arrays of bits) to represent which rows contain specific values. Efficient for columns with few distinct values.

116. What is storage optimization?
Techniques to reduce storage space and improve access speed: compression, partitioning, proper data types, etc.

117. What is query optimization?
The process of finding the most efficient way to execute a query—choosing indexes, join orders, and access methods.

118. What is buffer pool?
An area in memory (RAM) where the DBMS caches frequently accessed data pages to avoid slow disk reads.

119. What is buffer manager?
The component that manages the buffer pool—deciding which pages to keep in memory and which to write back to disk.

120. What is file manager?
The component that manages database files on disk—creating, deleting, reading, and writing files.

121. What is storage manager?
The overall component that handles all storage operations, including file management, buffer management, and space allocation.

122. What is page replacement?
When the buffer pool is full and a new page is needed, the DBMS must choose which existing page to remove (replace). Common algorithms: LRU (Least Recently Used).

123. What is compression?
Reducing the size of data stored on disk to save space and sometimes improve performance.

124. What is encryption?
Scrambling data so unauthorized users can’t read it, even if they access the physical files.

125. What is partitioning?
Splitting large tables into smaller pieces based on criteria (like date ranges) to improve performance and manageability.

126. What is clustering?
Storing related data physically close together on disk so it can be retrieved faster.

127. What is RAID?
Redundant Array of Independent Disks—using multiple disks together for better performance, capacity, or reliability.

128. What is tablespace?
A logical storage container that maps database objects (like tables) to physical files.

129. What are access paths?
The routes the DBMS can take to retrieve data: sequential scan, index scan, hash lookup, etc.

130. Why is physical layer hidden from users?
Users don’t need to know (and shouldn’t care) about storage details. This allows the DBMS to optimize storage independently and prevents users from causing damage.

SECTION 7 — DATA INDEPENDENCE

131. What is data independence?
The ability to change one level of the database (like physical storage) without affecting other levels (like applications).

132. Why is data independence important?
It allows the database to evolve and improve without breaking existing applications or requiring changes to user views.

133. What are types of data independence?
Two types:

  1. Physical Data Independence
  2. Logical Data Independence

134. What is physical data independence?
The ability to change physical storage (files, indexes, hardware) without changing the logical schema or applications.

135. What is logical data independence?
The ability to change the logical schema (add tables, change relationships) without changing external views or applications.

136. How does physical data independence work?
Example: You can switch from HDD to SSD, add indexes, or change file formats, and your SQL queries still work exactly the same.

137. How does logical data independence work?
Example: You can split one table into two for better design, but create a view that still looks like the original table to existing applications.

138. What changes are allowed in physical layer?

  • Changing storage devices
  • Adding/removing indexes
  • Changing file organizations
  • Compressing data
  • Partitioning tables
  • Changing RAID configurations

139. What changes are allowed in logical layer?

  • Adding new tables
  • Adding new columns (nullable)
  • Splitting or merging tables (with views)
  • Adding relationships
  • Changing constraints (carefully)

140. Why is logical data independence difficult?
Because applications often depend on specific table structures. Changing tables usually requires updating applications, unless views are used to mask changes.

141. Why is physical data independence easier?
Because applications and logical design don’t reference physical storage directly. The DBMS handles all physical details transparently.

142. How does data independence improve maintenance?
You can tune performance, upgrade hardware, or restructure data without disrupting users or rewriting applications.

143. How does data independence reduce coupling?
Applications don’t depend on physical storage details or specific logical structures—they depend on stable interfaces (views and APIs).

144. How does data independence support scalability?
You can scale storage (bigger disks, more servers) or optimize performance without application changes.

145. How does data independence support schema evolution?
The database schema can grow and change over time as business needs change, without breaking existing functionality.

SECTION 8 — SCHEMA VS INSTANCE

146. What is schema?
The blueprint or design of the database—tables, columns, relationships, and constraints. It’s the “structure.”

147. What is instance?
The actual data in the database at a specific moment—the “content” that changes over time.

148. Difference between schema and instance?

  • Schema: Structure, design, static (changes rarely)
  • Instance: Data, content, dynamic (changes constantly)

149. What is database state?
Same as instance—the current set of data in the database at a particular time.

150. What is intension?
Another term for schema—the intentional definition of the database structure.

151. What is extension?
Another term for instance—the extensional data that populates the schema.

152. Why is schema static?
Because the structure (tables, columns) doesn’t change with every insert or update. It changes only when you alter the database design.

153. Why is instance dynamic?
Because data is constantly being inserted, updated, and deleted by users and applications.

154. What does schema define?

  • What tables exist
  • What columns each table has
  • Data types of columns
  • Primary and foreign keys
  • Constraints and rules

155. What does instance represent?
The actual rows of data in tables at any given moment. Example: “John Doe, 25, john@email.com” is instance data.

156. What is relation schema?
The definition of a table: Student(StudentID INT, Name VARCHAR(50), Major VARCHAR(30))

157. What is relation instance?
The actual rows in that table:

StudentID Name Major
1 Alice CS
2 Bob Math

158. What is database blueprint?
The schema—the complete design that shows how the database is structured.

159. What is actual stored data?
The instance—all the rows and values currently in the database.

160. How frequently does schema change?
Rarely—only when the database design needs to change (adding features, restructuring).

161. How frequently does instance change?
Constantly—every time users add, update, or delete data.

162. What is time-dependent data?
Instance data, which represents the state of the database at a specific point in time.

163. What is time-independent definition?
Schema, which defines the structure regardless of what data is currently stored.

164. How are schema and instance related?
Every instance must follow the schema’s rules. The schema is the mold; the instance is what you pour into it.

165. Why must every instance obey schema?
Because the schema defines valid data. If the schema says Age must be a number, you can’t store “twenty-five” as Age.

SECTION 9 — RELATIONAL MODEL TERMINOLOGY

166. What is relation?
A table in the database. It has rows and columns and follows specific mathematical rules.

167. What is tuple?
A single row in a table. Example: (1, 'John', 'CS') is a tuple in a Student table.

168. What is attribute?
A column in a table. Example: StudentID, Name, and Major are attributes of the Student table.

169. What is domain?
The set of allowed values for an attribute. Example: Grade domain might be {A, B, C, D, F}.

170. What is degree of relation?
The number of columns (attributes) in a table. A table with 5 columns has degree 5.

171. What is cardinality of relation?
The number of rows (tuples) in a table. A table with 100 rows has cardinality 100.

172. What is candidate key?
A minimal set of columns that uniquely identifies each row. There can be multiple candidate keys in a table.

173. What is super key?
Any set of columns that uniquely identifies rows (may include extra unnecessary columns). Every candidate key is a super key, but not vice versa.

174. What is primary key?
The chosen candidate key that officially uniquely identifies rows in a table. Example: StudentID in a Student table.

175. What is composite key?
A key made of multiple columns together. Example: (CourseID, StudentID) might uniquely identify enrollments.

176. What is alternate key?
Candidate keys that were not chosen as primary key. Example: If you choose StudentID as primary key, Email (also unique) is an alternate key.

177. What is foreign key?
A column (or columns) in one table that references the primary key of another table, creating a relationship.

178. What is referential integrity?
The rule that foreign key values must match existing primary key values or be NULL. Prevents “orphan” records.

179. What is entity integrity?
The rule that primary keys cannot be NULL. Every row must be uniquely identifiable.

180. What is relational integrity?
The overall correctness and consistency of data in the relational database, enforced by all integrity constraints.

181. What is NULL value?
A special marker meaning “unknown” or “not applicable.” It’s not the same as zero or empty string.

182. What are relational constraints?
Rules that relations must follow: domain constraints, entity integrity, referential integrity, and user-defined constraints.

183. What is redundancy?
Storing the same data multiple times unnecessarily. Example: Storing department name with every employee instead of in a separate Department table.

184. What are update anomalies?
Problems caused by redundancy:

  • Insertion anomaly: Can’t insert data without other unrelated data
  • Update anomaly: Updating one copy but not others causes inconsistency
  • Deletion anomaly: Deleting data accidentally loses other information

185. What is normalization?
The process of organizing tables to minimize redundancy and avoid anomalies by following specific rules (normal forms).

SECTION 10 — CONSTRAINTS & INTEGRITY

186. What is constraint?
A rule that limits what data can be stored in the database to ensure accuracy and consistency.

187. Why are constraints important?
They prevent bad data from entering the database, maintain relationships between tables, and ensure business rules are followed.

188. What are integrity constraints?
Constraints that ensure data integrity—accuracy, consistency, and validity of data.

189. What is PRIMARY KEY constraint?
Ensures a column (or columns) uniquely identifies each row and cannot contain NULL values.

190. What is FOREIGN KEY constraint?
Ensures values in one table match values in another table, maintaining relationships between tables.

191. What is UNIQUE constraint?
Ensures all values in a column are different (but unlike primary key, allows one NULL).

192. What is NOT NULL constraint?
Ensures a column cannot contain NULL values—every row must have a value.

193. What is CHECK constraint?
Ensures values meet a specific condition. Example: CHECK (Age >= 18) ensures Age is 18 or older.

194. What is domain constraint?
Restricts values to a specific domain (data type and allowed values). Example: INT, VARCHAR, DATE.

195. What is referential integrity constraint?
Same as foreign key constraint—ensures relationships between tables remain valid.

196. What is entity integrity constraint?
Same as primary key constraint—ensures every row is uniquely identifiable.

197. What happens when constraints fail?
The DBMS rejects the operation (insert, update, or delete) and returns an error. The database remains unchanged.

198. What is constraint violation?
When you try to do something that breaks a constraint, like inserting a duplicate primary key or a foreign key that doesn’t exist.

199. What is domain violation?
When data doesn’t match the defined domain, like putting text in a number column or a date in the wrong format.

200. What is referential integrity violation?
When a foreign key value doesn’t match any primary key value in the referenced table, or when you try to delete a parent record that has children.

201. What is duplicate key violation?
When you try to insert a primary key or unique value that already exists in the table.

202. How does DBMS enforce constraints?
By checking every insert, update, and delete operation against the defined constraints and rejecting violations.

203. Why are constraints necessary?
Without constraints, databases would contain incorrect, inconsistent, and unreliable data—making them useless.

204. How do constraints maintain consistency?
By ensuring all data follows the same rules, relationships stay valid, and business logic is enforced automatically.

SECTION 11 — QUERY PROCESSING

205. What is query?
A request for data from the database, or a command to modify data.

206. What is SQL query?
A query written in SQL language. Example: SELECT * FROM Customers WHERE Country = 'USA'

207. What happens when SQL executes?

  1. Parsing (check syntax)
  2. Optimization (find best execution plan)
  3. Execution (run the plan)
  4. Return results

208. What is query parsing?
Checking if the SQL is valid, understanding what tables and columns are referenced, and building a parse tree.

209. What is query optimization?
Finding the fastest and most efficient way to execute a query by choosing indexes, join methods, and execution order.

210. What is execution plan?
The step-by-step strategy the DBMS creates to execute a query. You can view it to understand how the query will run.

211. What is query execution engine?
The component that carries out the execution plan and actually retrieves or modifies the data.

212. What is sequential scan?
Reading every row in a table one by one. Simple but slow for large tables.

213. What is index scan?
Using an index to find specific rows quickly without reading the entire table.

214. What is join algorithm?
A method for combining rows from two tables. Common types: Nested Loop Join, Hash Join, Merge Join.

215. How does optimizer choose plans?
It estimates the cost (time/resources) of different possible plans and picks the cheapest one based on statistics about the data.

216. What is query cost?
The estimated resources (CPU, I/O, memory) needed to execute a query. Lower cost = faster execution.

217. How does DBMS minimize I/O?
By using indexes, caching frequently accessed data in memory, and choosing efficient access paths.

218. How does indexing improve performance?
Like a book index—it lets the DBMS jump directly to relevant data instead of reading everything.

219. What is CPU cost in queries?
The processing power needed for calculations, sorting, comparisons, and other operations during query execution.

220. What is memory cost in queries?
The RAM needed to hold intermediate results, sort data, and cache information during query execution.

SECTION 12 — TRANSACTIONS & ACID

221. What is transaction?
A group of database operations that must all succeed or all fail together. Example: Transferring money involves deducting from one account and adding to another—both must happen or neither.

222. What is transaction manager?
The DBMS component that manages transactions—starting them, ensuring they complete properly, and handling failures.

223. What are ACID properties?
Four guarantees for reliable transactions:

  • Atomicity: All or nothing
  • Consistency: Database stays valid
  • Isolation: Transactions don’t interfere
  • Durability: Completed transactions survive crashes

224. What is atomicity?
A transaction is all-or-nothing. If any part fails, the entire transaction is rolled back (undone).

225. What is consistency?
A transaction brings the database from one valid state to another valid state, maintaining all constraints.

226. What is isolation?
Transactions execute as if they’re alone, even when running simultaneously. The result is the same as if they ran one at a time.

227. What is durability?
Once a transaction is committed (saved), it stays saved permanently, even if the system crashes immediately after.

228. What is concurrency control?
Managing multiple transactions happening at the same time to prevent conflicts and ensure isolation.

229. What is locking?
A concurrency control method where transactions lock data they’re using so other transactions can’t modify it until they’re done.

230. What is deadlock?
When two transactions are waiting for each other to release locks, creating a standstill. The DBMS detects and resolves this.

231. What is recovery system?
The mechanisms that restore the database to a consistent state after crashes, power failures, or errors.

232. How does DBMS recover after crash?
Using logs (records of all changes) to redo committed transactions and undo incomplete ones, bringing the database back to consistency.

233. How are concurrent transactions managed?
Through locking, timestamps, or versioning to ensure isolation while allowing multiple users to work simultaneously.

234. Why are transactions important?
They ensure data remains accurate and consistent even with failures, errors, or multiple users working at once.

235. What is rollback?
Undoing all changes made by a transaction when it fails or is cancelled, returning the database to its previous state.

236. What is commit?
Making all changes in a transaction permanent and visible to other users. After commit, changes are durable.

SECTION 13 — SECURITY & ACCESS CONTROL

237. What is database security?
Protecting data from unauthorized access, modification, or destruction through authentication, authorization, and encryption.

238. How do views improve security?
By restricting what data users can see. You can hide sensitive columns or rows by only granting access to specific views.

239. What is row-level security?
Restricting access to specific rows based on user identity. Example: Salespeople see only their own customers’ data.

240. What is column-level security?
Restricting access to specific columns. Example: Employees can see names but not salary information.

241. What is inference attack?
When someone deduces sensitive information from non-sensitive data. Example: Knowing someone is in a “High Salary” department reveals their salary range.

242. How does DBMS restrict access?
Through user accounts, passwords, roles, permissions (GRANT/REVOKE), views, and encryption.

243. What is authorization?
Giving users specific permissions to perform actions (read, write, delete) on specific database objects.

244. What is authentication?
Verifying who a user is—typically through usernames, passwords, or other credentials.

245. What is privilege management?
Controlling what actions each user or role can perform, and managing those permissions over time.

246. Why is abstraction important for security?
Because users interact with simplified views rather than raw data, reducing the attack surface and preventing direct access to sensitive information.

SECTION 14 — MODERN DATABASE CONCEPTS

247. What is ORM?
Object-Relational Mapping—a technique that lets developers work with databases using object-oriented programming instead of writing SQL directly.

248. What is object-relational mapping?
Same as ORM. It maps database tables to classes and rows to objects in programming languages.

249. What is impedance mismatch?
The difficulty of mapping between object-oriented programs (which use objects, inheritance, methods) and relational databases (which use tables, rows, columns).

250. What is NoSQL?
“Not Only SQL”—databases that don’t use the traditional relational model. Types include document stores (MongoDB), key-value stores (Redis), wide-column stores (Cassandra), and graph databases (Neo4j).

251. What is NewSQL?
Modern databases that combine the scalability of NoSQL with the ACID guarantees and SQL interface of relational databases. Example: Google Spanner, CockroachDB.

252. What is cloud database?
A database hosted on cloud platforms (AWS, Azure, Google Cloud) rather than on local servers, offering scalability and managed maintenance.

253. What is distributed database?
A database spread across multiple computers/servers, often in different locations, working together as a single system.

254. How do modern databases extend ANSI/SPARC?
By adding distributed layers, cloud abstractions, and multi-tenant views while maintaining the core three-level separation.

255. What is multi-tenancy?
When one database serves multiple customers (tenants) while keeping their data isolated and secure from each other.

256. How do cloud databases manage abstraction?
They hide infrastructure details (servers, networking, backups) and provide simple APIs and dashboards for users.

257. What are examples of distributed SQL databases?
CockroachDB, Google Spanner, YugabyteDB, TiDB.

258. How do ORMs interact with relational models?
They automatically convert between database tables/rows and programming objects, handling SQL generation behind the scenes.

SECTION 15 — REAL-WORLD IMPLEMENTATION

259. How does SQL implement ANSI/SPARC architecture?

  • External: Views in SQL
  • Conceptual: Tables, schemas, constraints
  • Internal: Storage engines, indexes, file formats

260. How does DBMS translate SQL internally?
SQL → Parser → Optimizer → Execution Plan → Storage Engine → Physical I/O → Results

261. How does logical schema map to physical storage?
Through the storage engine, which decides file formats, indexing, partitioning, and how tables are stored on disk.

262. How do queries travel through DBMS layers?
User Query → External (View resolution) → Conceptual (Table mapping) → Internal (Index/file selection) → Physical (Disk I/O)

263. How are tables converted into bytes?
The DBMS uses data type definitions to serialize each column value into binary format, stores rows in pages/blocks, and writes them to files.

264. How are rows stored physically?
As contiguous bytes in pages, often with headers (metadata) and footers (checksums), organized by the storage engine.

265. How are indexes maintained internally?
Using data structures like B+ Trees that are updated with every insert/update/delete to keep the mapping between key values and row locations current.

266. How does DBMS enforce constraints internally?
By checking every data modification against constraint definitions, often using indexes for uniqueness checks and foreign key validations.

267. How do views hide schema complexity?
By presenting a simplified interface that combines tables, filters data, and renames columns, so users don’t see the underlying complexity.

268. How do modern DBMSs implement abstraction?
Through pluggable storage engines, query optimizers, view systems, and APIs that separate user interfaces from physical implementation.

269. How does DBMS maintain independence between layers?
Through mappings that translate requests between levels, ensuring changes at one level don’t require changes at others.

270. How does abstraction help real-world scalability?
You can scale hardware, add servers, optimize storage, and restructure data without disrupting applications or users—critical for growing businesses.
