I’ll answer all 270 database questions in a clear, detailed, and easy-to-understand way. This is comprehensive, so I’ll organize it by section with practical examples.

SECTION 1 — DATABASE FUNDAMENTALS

1. What is a database?

A database is an organized collection of data stored electronically. Think of it as a digital filing cabinet where information is structured so you can easily find, add, update, or delete data. Example: A university database stores student records, courses, and grades.

2. What is a DBMS?

A Database Management System (DBMS) is software that lets you create, manage, and interact with databases. It’s like the operating system for your data. Examples: MySQL, PostgreSQL, Oracle, MongoDB.

3. Why do we need DBMS?

Without a DBMS, you’d store data in plain files (like Excel or text files), which causes problems:

  • Data redundancy: Same data copied in multiple places
  • Inconsistency: Updating one copy but not others
  • No concurrent access: Two people can’t edit safely at once
  • No crash recovery: Lose everything if computer crashes
  • Security issues: Can’t easily control who sees what

4. What problems exist in file systems?

File systems have these major issues:

  • Data isolation: Data scattered across many files
  • Duplication: Same customer address in 10 files
  • Dependency: Programs are written for specific file formats
  • No integrity checks: Can enter invalid data (like age = -5)
  • No atomic operations: A transfer might debit one account but not credit another

5. How does DBMS manage data?

A DBMS manages data through:

  • Storage management: Decides where and how to store data on disk
  • Query processing: Interprets your requests and finds data efficiently
  • Transaction management: Ensures operations complete properly
  • Concurrency control: Handles multiple users simultaneously
  • Recovery: Restores data after crashes
  • Security: Controls who can access what

6. What are responsibilities of DBMS?

  • Data definition (creating tables/structures)
  • Data manipulation (insert, update, delete, query)
  • Data security and integrity enforcement
  • Multi-user access control
  • Backup and recovery
  • Performance optimization

7. What are examples of DBMS?

  • Relational: MySQL, PostgreSQL, Oracle, SQL Server, SQLite
  • NoSQL: MongoDB (document), Redis (key-value), Cassandra (column), Neo4j (graph)
  • Cloud: Amazon RDS, Google Cloud Spanner, Azure SQL

8. What is relational DBMS?

A Relational DBMS (RDBMS) stores data in tables (relations) with rows and columns. Tables can relate to each other through keys. It’s based on mathematical set theory. Example: A Students table links to an Enrollments table via student_id.

9. What is SQL?

SQL (Structured Query Language) is the standard language for talking to relational databases. You use it to ask questions (queries) and give commands.

SELECT name FROM students WHERE grade = 'A';

10. Why is SQL important?

  • Universal: Works across almost all relational databases
  • Declarative: You say what you want, not how to get it
  • Standardized: ANSI/ISO standard since 1986
  • Powerful: Handles complex data operations simply
  • Industry standard: Required skill for data jobs

SECTION 2 — DATA ABSTRACTION

11. What is data abstraction?

Data abstraction means hiding complex details and showing only essential information. Like driving a car—you use the steering wheel and pedals without knowing how the engine works internally.

12. Why is data abstraction needed?

Because databases are incredibly complex internally. Users shouldn’t need to understand disk sectors, memory buffers, indexing algorithms, or query optimization just to retrieve customer names.

13. What complexity does DBMS hide?

  • Physical storage details (where bytes sit on disk)
  • Index structures (B+ trees, hash tables)
  • Query optimization (choosing the fastest execution plan)
  • Memory management (buffer pools, caching)
  • Concurrency control (locking mechanisms)
  • Recovery procedures (logging, checkpoints)

14. How does abstraction simplify databases?

Users interact with a simplified model: tables, rows, and SQL commands. The DBMS handles the translation to physical operations automatically.

15. What are advantages of abstraction?

  • Easier to learn and use
  • Less error-prone
  • Changes to internals don’t break applications
  • Multiple users see customized views
  • Security through information hiding

16. What problems happen without abstraction?

  • Every programmer must understand disk I/O, memory, indexing
  • Applications break when storage changes
  • No security boundaries
  • Code becomes unmaintainable
  • Users accidentally corrupt data

17. How does abstraction improve security?

By showing users only what they need. A payroll clerk sees salaries but not server passwords. A manager sees department summaries, not individual employee records.

18. How does abstraction improve maintainability?

When internal storage changes (e.g., moving from HDD to SSD), only the DBMS internal code changes. Applications using SQL continue working unchanged.

19. How does abstraction improve flexibility?

You can change the physical layout (add indexes, partition tables) without changing the logical design or user applications.

20. How does abstraction improve usability?

Users write SELECT * FROM customers instead of writing code to traverse disk blocks, parse page headers, and assemble records.

21. What is the real-world meaning of abstraction?

Like using a TV remote—you press “power” and the TV turns on. You don’t need to understand electricity, circuit boards, or signal processing.

22. How is abstraction used in DBMS?

Through three levels: External (user views), Conceptual (logical design), Internal (physical storage). Each level hides the one below it.

23. What is separation of concerns?

Dividing a system into distinct sections, each handling one aspect. The DBMS separates: user interface, data logic, and storage mechanics.

24. How does DBMS separate complexity into layers?

  • External layer: What users see (views)
  • Conceptual layer: What the database logically contains (tables, relationships)
  • Internal layer: How it’s physically stored (files, indexes, blocks)

25. What happens internally when user executes SQL?

  1. Parse: Check syntax and semantics
  2. Optimize: Find the cheapest execution plan
  3. Compile: Generate execution code
  4. Execute: Run against storage, using indexes if helpful
  5. Return: Format and send results to user

26. What internal operations are hidden from users?

  • Parsing and validating the SQL statement
  • Rewriting the query for efficiency
  • Choosing between index scan vs. table scan
  • Locking rows to prevent conflicts
  • Reading/writing disk blocks
  • Managing buffer pool memory
  • Writing to transaction logs

27. How does DBMS hide storage details?

Users see “tables.” The DBMS decides whether to store rows in one file or many, compressed or uncompressed, on SSD or in memory.

28. How does DBMS hide indexing details?

Users don’t manually create or navigate indexes. The DBMS automatically uses indexes when beneficial, and users don’t even know indexes exist unless they create them.

29. How does DBMS hide query optimization?

The DBMS might rewrite your query, change join order, or use an index—all automatically. You write simple SQL; the DBMS finds the fastest way to run it.

30. Why should users not interact with physical storage directly?

  • They could corrupt data structures
  • They’d need specialized knowledge
  • Applications would become tied to specific hardware
  • Security would be impossible to enforce

SECTION 3 — ANSI/SPARC ARCHITECTURE

31. What is ANSI/SPARC architecture?

A standard framework (1975) proposed by ANSI (American National Standards Institute) and SPARC (Standards Planning and Requirements Committee) defining three levels of database abstraction.

32. Why was ANSI/SPARC introduced?

Before this, database systems were inconsistent. There was no standard way to talk about data independence or layered design. This framework provided a common vocabulary and structure.

33. What problems existed before ANSI/SPARC?

  • Applications were tightly coupled to physical storage
  • Changing storage format broke all programs
  • No clean separation between user views and database design
  • Every DBMS had a different architecture

34. What are the three levels of DBMS architecture?

  1. External Level (User views)
  2. Conceptual Level (Logical structure)
  3. Internal Level (Physical storage)

35. What is the purpose of 3-level architecture?

To achieve data independence—the ability to change one level without affecting others.

36. How does ANSI/SPARC architecture work?

Like an office building:

  • Lobby (External): Different visitors see different areas
  • Floor plans (Conceptual): The actual layout of rooms
  • Foundation (Internal): Concrete, wiring, plumbing

37. What is abstraction hierarchy in DBMS?

A pyramid where each layer builds on the one below:

  • Top: External views (most abstract, user-specific)
  • Middle: Conceptual schema (logical design)
  • Bottom: Internal schema (physical implementation)

38. Why is layered DBMS design important?

  • Isolates changes
  • Supports multiple user perspectives
  • Allows optimization at each level independently
  • Enables security boundaries

39. How do abstraction levels communicate?

Through mappings (translations). When a user queries a view, the DBMS maps it to the conceptual schema, then to the internal schema, then executes on physical storage.

40. What is schema mapping?

The translation process between levels. It’s like translating from English (external) to French (conceptual) to German (internal).

41. What is external-conceptual mapping?

Translates a user’s view into the conceptual schema. If a view shows employee_name but the conceptual table calls it full_name, this mapping handles the conversion.

42. What is conceptual-internal mapping?

Translates the logical design into physical storage instructions. The conceptual Employees table might be stored in file emp.data with B+ tree index emp.idx.

43. How does data move between layers?

Downward: User query → External view → Conceptual schema → Internal schema → Physical I/O
Upward: Physical data → Internal format → Conceptual format → External view → User display

44. What are advantages of layered architecture?

  • Data independence: Change physical storage without touching applications
  • Multiple views: Different users see different data
  • Security: Hide sensitive data from unauthorized users
  • Simplification: Each layer deals with manageable complexity

45. What are disadvantages of layered architecture?

  • Performance overhead: Multiple translations take time
  • Complexity: More layers = more code to maintain
  • Storage: Mappings themselves need storage and management

46. How does layered architecture improve scalability?

Change physical storage (add servers, shard data) without rewriting applications. The conceptual layer remains stable.

47. How does layered architecture improve maintainability?

Fix a bug in the storage layer, and all applications continue working. Update a view definition without touching the underlying tables.

48. How does layered architecture improve security?

Create views that exclude sensitive columns. Users interact only with views, never seeing the base tables or physical files.

49. How does layered architecture improve portability?

Move from Oracle to PostgreSQL—applications using standard SQL need minimal changes because the abstraction layers handle differences.

50. How does layered architecture improve optimization?

The DBMS can change physical organization (add indexes, partition tables) to improve speed without requiring application changes.

SECTION 4 — EXTERNAL / VIEW LEVEL

51. What is external level?

The top layer of the ANSI/SPARC architecture. It’s what individual users or applications see—their personalized window into the database.

52. What is view level?

Same as external level. “View” because it’s a particular way of looking at the data, like looking through colored glass that shows only certain colors.

53. What is external schema?

The definition of a specific user view. It describes what tables, columns, and rows a particular user can see.

54. What is view schema?

Same as external schema—the structure definition of a view.

55. What is subschema?

An older term for external schema. In CODASYL/network databases, the subschema defined which parts of the database a program could access.

56. Why do we need external views?

  • Security: Hide sensitive data (salaries, passwords)
  • Simplicity: Show only relevant data to each job role
  • Customization: Rename columns for user convenience
  • Data independence: Protect apps from schema changes

57. What is user-specific data representation?

A view tailored to a user’s needs. A sales rep sees their customers and orders; an accountant sees invoices and payments.

58. How do different users see different data?

The DBMS creates different views for different users. Same underlying database, different filtered perspectives.

-- Manager sees all employees
CREATE VIEW manager_employees AS SELECT * FROM employees;

-- Employee sees only their own record
CREATE VIEW my_record AS SELECT * FROM employees WHERE emp_id = CURRENT_USER;

59. How do views improve security?

By restricting what users can see. Even if they try to access other data, the view definition prevents it.

60. What is view-level security?

Security enforced through views—granting access to views rather than base tables. Users can only see what the view allows.

61. What is row filtering?

A view that shows only certain rows. Example: WHERE department = 'Sales' shows only sales employees.

62. What is column filtering?

A view that hides certain columns. Example: Excluding salary and ssn columns from a view shown to general employees.

63. What is customized data access?

Creating views that join tables, compute values, or rename columns to match user expectations or application needs.

64. What is data hiding using views?

Concealing the existence or values of certain data. Users don’t know what they’re missing—it’s invisible to them.

65. What are examples of external views?

  • A view showing only active customers for the sales team
  • A view aggregating monthly sales for managers
  • A view joining employee and department data for HR
  • A view excluding financial data for public access

66. What is virtual table?

A view is called a virtual table because it looks and acts like a table but doesn’t physically store data. It’s defined by a query that runs when you access it.

67. What is a database view?

A saved query that presents data as if it were a table. It derives its data from base tables (or other views) in real-time.

68. What is materialized view?

A view that actually stores the query results physically, like caching. It’s refreshed periodically. Faster to read, slower to update.

69. Difference between virtual view and materialized view?

Aspect Virtual View Materialized View
Storage No physical storage Stores actual data
Freshness Always current May be stale
Read speed Slower (query runs each time) Faster (pre-computed)
Write impact None Needs refresh
Space Minimal Uses disk space

70. What is view abstraction?

The principle that views provide an abstract, simplified interface to complex underlying data structures.

71. Can multiple views exist for same database?

Yes—hundreds or thousands. Each user or application can have its own view of the same underlying data.

72. What users interact with external level?

  • End users through applications
  • Report generators
  • Business analysts
  • Any application programmer using the database

73. How do applications use views?

Applications query views as if they were tables. The view insulates the app from underlying schema changes.

74. How does external layer simplify databases?

Users don’t need to understand complex joins, table relationships, or normalization. They see a flat, relevant dataset.

75. What is computed column in views?

A column calculated from other columns in the view definition.

CREATE VIEW order_summary AS
SELECT order_id, quantity, price, quantity * price AS total_amount
FROM order_items;

76. What is aggregation in views?

Using GROUP BY and aggregate functions to summarize data in a view.

CREATE VIEW monthly_sales AS
SELECT MONTH(order_date) AS month, SUM(amount) AS total
FROM orders
GROUP BY MONTH(order_date);

77. What is view updatability?

Whether you can INSERT, UPDATE, or DELETE through a view. Simple views on single tables usually allow this; complex views with joins/aggregations often don’t.

78. What are limitations of views?

  • Performance: Complex views may run slowly
  • Updatability: Not all views support modifications
  • Dependencies: Changing base tables can break views
  • Nesting: Views on views on views become hard to debug

SECTION 5 — CONCEPTUAL / LOGICAL LEVEL

79. What is conceptual level?

The middle layer describing the entire database’s logical structure—what data exists, how it’s organized, and what rules apply. Independent of physical storage and user views.

80. What is logical level?

Same as conceptual level. It deals with the logical (not physical) organization of data.

81. What is conceptual schema?

The complete logical design of the database—all tables, columns, relationships, constraints, and business rules. The “master blueprint.”

82. What is logical schema?

Same as conceptual schema—the structural definition at the logical level.

83. What does conceptual level describe?

  • All entities (things we store data about)
  • Their attributes (properties)
  • Relationships between entities
  • Integrity constraints (rules data must follow)
  • Business rules

84. What exists at conceptual level?

Tables, columns, primary keys, foreign keys, constraints, domains, and relationships. The complete database design.

85. What is logical database structure?

The organization of data into tables and relationships, independent of how it’s stored on disk.

86. What are entities?

Real-world objects or concepts we store information about. Examples: Student, Course, Employee, Product.

87. What are attributes?

Properties or characteristics of entities. A Student has attributes: name, student_id, email, major.

88. What are relationships?

How entities connect to each other. A Student enrolls in a Course. An Employee works in a Department.

89. What are constraints?

Rules that data must obey to be valid. Example: Age must be positive. Every student must have a unique ID.

90. What are business rules?

Organization-specific policies expressed as database constraints. “Every order must have a valid customer.” “No employee can manage more than 10 people.”

91. What are domains?

The set of valid values for an attribute. The domain of “Gender” might be {M, F, Other}. The domain of “Grade” might be {A, B, C, D, F}.

92. What is relation schema?

The definition of a table: its name, columns, and their data types and constraints.

Students(student_id INT PRIMARY KEY, name VARCHAR(50), major VARCHAR(30))

93. How are tables represented logically?

As relations (mathematical sets of tuples). Each row is a tuple; each column is an attribute.

94. What is logical organization of data?

Grouping related data into tables and establishing relationships through keys, without worrying about disk blocks or file formats.

95. What is complete database blueprint?

The conceptual schema showing all tables, columns, keys, and relationships. Database administrators and designers work at this level.

96. What is schema design?

The process of creating the conceptual schema—deciding what tables to create, what columns they need, and how they relate.

97. What is entity relationship modeling?

A design technique using diagrams (ER diagrams) to plan databases. Boxes represent entities, lines represent relationships.

98. What is logical data modeling?

Creating a detailed model of data structures, relationships, and constraints before implementing them in a specific DBMS.

99. What users work at logical level?

  • Database designers
  • Database administrators (DBAs)
  • Data architects
  • Senior developers designing data structures

100. Why is conceptual level important?

It’s the stable foundation of the database. Changes here affect everyone, so it must be designed carefully. It represents the “truth” of the organization’s data.

SECTION 6 — INTERNAL / PHYSICAL LEVEL

101. What is internal level?

The bottom layer of ANSI/SPARC. It describes how data is actually stored on physical hardware—files, indexes, blocks, compression, encryption.

102. What is physical level?

Same as internal level. Concerned with bits, bytes, disk sectors, and memory.

103. What is physical schema?

The definition of how conceptual objects map to physical storage—file names, record formats, indexing methods, compression techniques.

104. What is storage schema?

Same as physical schema—the blueprint for physical storage.

105. What does internal level describe?

  • File organization (how records are arranged in files)
  • Index structures (B+ trees, hash tables)
  • Storage allocation (disk blocks, pages)
  • Compression and encryption methods
  • Buffer management

106. What exists at physical level?

Data files, index files, log files, temporary files, system catalogs, and the actual bytes on disk.

107. How is data physically stored?

In operating system files or raw disk partitions, organized into pages/blocks (typically 4KB, 8KB, or 16KB each).

108. What are disk blocks?

The smallest unit of disk I/O. The OS and DBMS read/write in block-sized chunks (e.g., 8KB at a time).

109. What are pages in DBMS?

Same as disk blocks—the fixed-size units of storage that the DBMS manages in memory and on disk.

110. What is file organization?

How records are arranged in a file:

  • Heap: Unordered, append-only
  • Sorted: Ordered by some key
  • Hash: Organized by hash function
  • Indexed: With separate index structures

111. What are indexes?

Data structures that speed up data retrieval, like the index at the back of a book. They map key values to record locations.

112. What is indexing?

The technique of creating and maintaining index structures to avoid scanning entire tables.

113. What is B+ Tree?

A balanced tree data structure optimized for disk storage. All data pointers are in leaf nodes, which are linked together. Very efficient for range queries.

114. What is hash indexing?

Using a hash function to map keys directly to storage locations. Extremely fast for exact-match lookups, useless for ranges.

115. What is bitmap indexing?

Using bit arrays to represent which rows contain specific values. Excellent for low-cardinality columns (gender, status, boolean flags) in data warehouses.

116. What is storage optimization?

Techniques to reduce space and improve speed: compression, deduplication, columnar storage, partitioning.

117. What is query optimization?

The DBMS process of finding the cheapest/fastest way to execute a query—choosing join order, access methods, and execution strategy.

118. What is buffer pool?

A region of memory where the DBMS caches disk pages. Recently used data stays in RAM for fast access.

119. What is buffer manager?

The DBMS component that manages the buffer pool—deciding which pages to keep in memory and which to evict when space is needed.

120. What is file manager?

The component that handles reading/writing pages to disk files, managing file growth, and allocating disk space.

121. What is storage manager?

The overall subsystem handling all physical storage operations—file manager, buffer manager, and space allocation.

122. What is page replacement?

When the buffer pool is full and a new page is needed, the buffer manager chooses which existing page to remove (evict). Algorithms: LRU (Least Recently Used), MRU, Clock.

123. What is compression?

Reducing data size using algorithms (like gzip, dictionary compression) to save disk space and I/O bandwidth.

124. What is encryption?

Scrambling data so unauthorized users can’t read it, even if they access the physical files. Requires keys to decrypt.

125. What is partitioning?

Splitting large tables into smaller pieces based on criteria (range, list, hash). Improves query performance and maintenance.

126. What is clustering?

Storing related rows physically close together. Clustering by customer_id keeps all orders for one customer on the same disk pages.

127. What is RAID?

Redundant Array of Independent Disks. Combining multiple disks for performance (striping) or reliability (mirroring). RAID 0 = speed, RAID 1 = mirror, RAID 5/6 = parity.

128. What is tablespace?

A logical container for database objects (tables, indexes). Maps to physical files. Allows organizing storage by function or performance needs.

129. What are access paths?

The routes to retrieve data: sequential scan (read all rows), index scan (use index), index-only scan (get data from index without touching table).

130. Why is physical layer hidden from users?

Users would be overwhelmed by complexity. More importantly, it allows the DBMS to change storage technology (HDD→SSD→memory) without breaking applications.

SECTION 7 — DATA INDEPENDENCE

131. What is data independence?

The ability to change one schema level without having to change the levels above it. The holy grail of database design.

132. Why is data independence important?

It decouples applications from data structures, allowing evolution and optimization without massive rewrites.

133. What are types of data independence?

Two types:

  • Physical data independence
  • Logical data independence

134. What is physical data independence?

Ability to change the physical schema without changing the conceptual schema. You can add indexes, change file organizations, or move to faster disks without touching SQL queries.

135. What is logical data independence?

Ability to change the conceptual schema without changing external schemas or applications. You can add a column to a table without breaking existing views.

136. How does physical data independence work?

Through the conceptual-internal mapping. When physical storage changes, only this mapping is updated. The conceptual schema remains the same, so all applications continue working.

137. How does logical data independence work?

Through the external-conceptual mapping. When the conceptual schema changes (e.g., splitting a table), views can be adjusted to present the old structure, keeping applications unchanged.

138. What changes are allowed in physical layer?

  • Adding/dropping indexes
  • Changing file organizations
  • Moving data to different disks
  • Compressing data
  • Changing page sizes
  • Using different RAID configurations

139. What changes are allowed in logical layer?

  • Adding new tables or columns
  • Splitting or merging tables
  • Adding relationships
  • Changing constraints (carefully)
  • Creating new views

140. Why is logical data independence difficult?

Because applications often depend on specific table structures. Adding a column is easy; removing a column or changing its meaning can break applications. Views help but can’t hide all changes.

141. Why is physical data independence easier?

Because the conceptual schema insulates applications. As long as tables and columns remain logically the same, physical changes are invisible.

142. How does data independence improve maintenance?

DBAs can tune performance (add indexes, partition tables) without involving developers. Developers can evolve schemas without breaking all applications at once.

143. How does data independence reduce coupling?

Applications depend on logical structures (views, stable table designs) rather than physical details. Changes in one area don’t cascade to others.

144. How does data independence support scalability?

You can scale storage (add servers, shard data) or optimize performance at the physical level while keeping the application layer stable.

145. How does data independence support schema evolution?

Organizations change over time. Data independence lets the database evolve (new fields, new tables, new relationships) without requiring every application to be rewritten simultaneously.

SECTION 8 — SCHEMA VS INSTANCE

146. What is schema?

The blueprint or definition of a database—tables, columns, types, constraints. It’s the structure or intension. Static and rarely changes.

147. What is instance?

The actual data in the database at a specific moment—the rows in the tables. Dynamic and changes constantly.

148. Difference between schema and instance?

Schema Instance
Definition/structure Actual data content
Static (rarely changes) Dynamic (changes frequently)
Created by DDL (CREATE TABLE) Modified by DML (INSERT, UPDATE)
Blueprint Building contents

149. What is database state?

The complete set of data in the database at a specific point in time. Same as “instance.”

150. What is intension?

The formal term for schema—the intentional definition of what the database contains.

151. What is extension?

The formal term for instance—the extensional set of actual tuples/rows.

152. Why is schema static?

Because changing structure (adding tables, modifying columns) requires careful planning and can break applications. It’s a deliberate design activity.

153. Why is instance dynamic?

Because business operations constantly create, update, and delete data. Every new order, customer registration, or inventory change updates the instance.

154. What does schema define?

  • What tables exist
  • What columns each table has
  • Data types and constraints
  • Relationships and keys
  • Indexes and views

155. What does instance represent?

The current reality—the actual customers, orders, products, and transactions stored in the database right now.

156. What is relation schema?

The definition of a table: Employee(emp_id, name, salary, dept_id) with types and constraints.

157. What is relation instance?

The actual set of rows in that table at a moment: {(1, ‘Alice’, 50000, 10), (2, ‘Bob’, 60000, 20), …}.

158. What is database blueprint?

The schema—the architectural plan showing all structures and rules.

159. What is actual stored data?

The instance—the population of the database with real values.

160. How frequently does schema change?

Rarely. Schema changes (migrations) happen during development, upgrades, or major business changes—maybe monthly or quarterly in production.

161. How frequently does instance change?

Constantly. Every transaction modifies the instance—potentially thousands of times per second in busy systems.

162. What is time-dependent data?

Data whose values change over time. The instance captures a snapshot, but real-world data evolves.

163. What is time-independent definition?

The schema doesn’t change when data changes. The definition of “Employee” remains stable even as employees come and go.

164. How are schema and instance related?

Every instance must conform to the schema. You can’t have a row with columns that don’t exist in the schema, or values that violate constraints.

165. Why must every instance obey schema?

Because the schema defines the rules. Violations would mean data corruption, inconsistency, and broken applications. The DBMS enforces this obedience.

SECTION 9 — RELATIONAL MODEL TERMINOLOGY

166. What is relation?

A table—a set of tuples (rows) with the same structure. Mathematically, a subset of the Cartesian product of domains.

167. What is tuple?

A row in a table. An ordered set of values, one for each attribute.

168. What is attribute?

A column in a table. A named property of a relation.

169. What is domain?

The set of permissible values for an attribute. Like the data type but can be more specific (e.g., integers between 1 and 100).

170. What is degree of relation?

The number of attributes (columns) in a relation. A table with 5 columns has degree 5.

171. What is cardinality of relation?

The number of tuples (rows) in a relation. Changes as data is inserted/deleted.

172. What is candidate key?

A minimal set of attributes that uniquely identifies each tuple. A table may have multiple candidate keys.

Example: In a Students table, both student_id and email might be unique—both are candidate keys.

173. What is super key?

Any set of attributes that uniquely identifies tuples (not necessarily minimal). A candidate key is a minimal super key. {student_id, name} is a super key but not a candidate key because name is redundant.

174. What is primary key?

The candidate key chosen by the designer as the main identifier. Every table should have one. Example: student_id is declared PRIMARY KEY.

175. What is composite key?

A key consisting of multiple attributes. Example: In an Enrollments table, (student_id, course_id) together form the primary key.

176. What is alternate key?

A candidate key that is NOT chosen as the primary key. If email is unique but student_id is primary key, email is an alternate key.

177. What is foreign key?

An attribute (or set) in one table that references the primary key of another table. Establishes relationships.

-- dept_id in Employees references id in Departments
FOREIGN KEY (dept_id) REFERENCES Departments(id)

178. What is referential integrity?

The rule that foreign key values must either match a primary key value in the referenced table or be NULL. Prevents “dangling references.”

179. What is entity integrity?

The rule that primary keys cannot be NULL. Every row must be uniquely identifiable.

180. What is relational integrity?

The overall correctness of data with respect to all defined constraints (entity integrity, referential integrity, domain constraints, user-defined constraints).

181. What is NULL value?

A special marker indicating “unknown” or “not applicable.” Not the same as zero or empty string. NULL + 5 = NULL.

182. What are relational constraints?

Rules defining valid states:

  • Domain constraints: Values must be from declared domains
  • Key constraints: Primary and candidate keys must be unique
  • Referential integrity: Foreign keys must be valid
  • Entity integrity: Primary keys can’t be NULL

183. What is redundancy?

Storing the same information multiple times. Wastes space and causes update anomalies.

184. What are update anomalies?

Problems caused by redundancy:

  • Insertion anomaly: Can’t insert data because other required (unrelated) data is missing
  • Deletion anomaly: Deleting one fact accidentally deletes another
  • Update anomaly: Changing one copy but not others causes inconsistency

185. What is normalization?

The process of organizing tables to minimize redundancy and eliminate anomalies. Divides large tables into smaller, well-structured ones.

SECTION 10 — CONSTRAINTS & INTEGRITY

186. What is constraint?

A rule enforced by the DBMS that limits what data can be stored. Ensures correctness and consistency.

187. Why are constraints important?

They prevent invalid data, maintain business rules automatically, and ensure data quality without relying on application code.

188. What are integrity constraints?

Constraints that preserve data correctness and consistency. The “integrity” of the database.

189. What is PRIMARY KEY constraint?

Ensures a column (or combination) uniquely identifies each row and cannot be NULL.

CREATE TABLE students (
    student_id INT PRIMARY KEY,
    name VARCHAR(50)
);

190. What is FOREIGN KEY constraint?

Ensures values in a column match values in a referenced table’s primary key, maintaining referential integrity.

191. What is UNIQUE constraint?

Ensures no duplicate values in a column (unlike PRIMARY KEY, allows one NULL).

192. What is NOT NULL constraint?

Prevents NULL values in a column. Every row must have a value.

193. What is CHECK constraint?

Validates data against a boolean expression.

CREATE TABLE employees (
    age INT CHECK (age >= 18 AND age <= 65),
    salary DECIMAL CHECK (salary > 0)
);

194. What is domain constraint?

Restricts values to a specific data type or set. The domain of “DayOfWeek” might be {Mon, Tue, Wed, Thu, Fri, Sat, Sun}.

195. What is referential integrity constraint?

The rule enforced by FOREIGN KEY constraints: no orphaned references.

196. What is entity integrity constraint?

The rule enforced by PRIMARY KEY constraints: no NULL primary keys, no duplicates.

197. What happens when constraints fail?

The DBMS rejects the operation and returns an error. The transaction may be rolled back.

198. What is constraint violation?

Attempting an operation that breaks a constraint. Example: Inserting duplicate primary key, or a foreign key that doesn’t exist.

199. What is domain violation?

Inserting a value outside the permitted domain. Example: Putting ‘ABC’ in an INTEGER column, or -5 in an AGE column.

200. What is referential integrity violation?

Inserting a row with a foreign key value that doesn’t exist in the referenced table, or deleting a referenced row that still has foreign keys pointing to it.

201. What is duplicate key violation?

Trying to insert or update a primary key or unique key to a value that already exists.

202. How does DBMS enforce constraints?

  • Immediate checking: Validates on every INSERT/UPDATE
  • Deferred checking: Validates at transaction commit (for complex cases)
  • Uses indexes for fast uniqueness checks
  • Uses triggers and rules for complex constraints

203. Why are constraints necessary?

Application bugs, user errors, and concurrent operations can all introduce bad data. Constraints are the last line of defense—enforced by the DBMS itself, impossible to bypass.

204. How do constraints maintain consistency?

By rejecting invalid changes before they become permanent. They ensure the database always satisfies business rules, even when applications have bugs.

SECTION 11 — QUERY PROCESSING

205. What is query?

A request for data or an operation on data. “Give me all customers from New York” or “Increase all prices by 10%.”

206. What is SQL query?

A query written in SQL language.

SELECT name, email FROM customers WHERE city = 'New York';

207. What happens when SQL executes?

  1. Parse and validate
  2. Optimize (find best execution plan)
  3. Execute (access storage, process data)
  4. Return results

208. What is query parsing?

Checking that the SQL is syntactically correct and semantically valid (tables/columns exist, user has permissions).

209. What is query optimization?

Finding the most efficient way to execute a query. The DBMS considers many possible plans and picks the cheapest.

210. What is execution plan?

The step-by-step strategy the optimizer chooses. Shows which indexes to use, join order, and access methods.

211. What is query execution engine?

The DBMS component that carries out the execution plan, coordinating storage access, joins, sorting, and aggregation.

212. What is sequential scan?

Reading every row in a table, one after another. Simple but slow for large tables. Used when no index helps or when most rows will be returned anyway.

213. What is index scan?

Using an index to find specific rows quickly. Jump directly to relevant data instead of reading the whole table.

214. What is join algorithm?

Methods for combining rows from multiple tables:

  • Nested Loop: For each row in table A, scan table B
  • Hash Join: Build hash table from one table, probe with the other
  • Merge Join: Sort both tables, then merge (efficient for large sorted data)

215. How does optimizer choose plans?

Estimates cost (disk I/O, CPU, memory) for each possible plan using statistics (table sizes, index selectivity, data distribution). Picks the cheapest.

216. What is query cost?

The estimated resource consumption—primarily disk I/O operations, plus CPU and memory usage. Lower cost = faster execution.

217. How does DBMS minimize I/O?

  • Uses indexes to avoid full table scans
  • Caches frequently accessed pages in buffer pool
  • Prefetches pages it expects to need
  • Uses covering indexes (index contains all needed columns)

218. How does indexing improve performance?

Instead of reading 1 million rows to find 10 matches, an index lets the DBMS jump directly to those 10 rows. Can speed up queries by 1000x or more.

219. What is CPU cost in queries?

Processing time for operations like comparisons, calculations, sorting, and hashing. Less significant than I/O cost but matters for complex queries.

220. What is memory cost in queries?

RAM needed for sorting, hashing, and caching. Large sorts or joins may require spilling to disk if memory is insufficient, slowing execution dramatically.

SECTION 12 — TRANSACTIONS & ACID

221. What is transaction?

A logical unit of work—a sequence of operations that must complete entirely or not at all. Example: Transferring $100 from Account A to Account B involves two updates; both must succeed or both must fail.

222. What is transaction manager?

The DBMS component that oversees transactions—starting them, tracking their operations, and ensuring they complete properly or are undone.

223. What are ACID properties?

The four guarantees of reliable transaction processing:

  • Atomicity
  • Consistency
  • Isolation
  • Durability

224. What is atomicity?

“All or nothing.” A transaction’s operations either all complete successfully, or all are undone (rolled back). No partial completion.

225. What is consistency?

A transaction brings the database from one valid state to another. All constraints and rules remain satisfied after the transaction.

226. What is isolation?

Concurrent transactions don’t interfere with each other. Each transaction executes as if it’s the only one running, even when many run simultaneously.

227. What is durability?

Once a transaction commits, its changes are permanent—even if the system crashes immediately after. Usually achieved by writing to transaction logs before confirming commit.

228. What is concurrency control?

Managing simultaneous transactions so they don’t corrupt each other’s work while maintaining good performance.

229. What is locking?

A concurrency control method where transactions acquire locks on data before accessing it. Other transactions must wait.

  • Shared lock (S): For reading—multiple readers allowed
  • Exclusive lock (X): For writing—no other access allowed

230. What is deadlock?

When two transactions each hold a lock the other needs, and both wait forever. Example:

  • Transaction A locks Row 1, wants Row 2
  • Transaction B locks Row 2, wants Row 1

Solution: Deadlock detection (DBMS kills one) or timeout.

231. What is recovery system?

The DBMS subsystem that restores the database to a consistent state after crashes, using transaction logs and checkpoints.

232. How does DBMS recover after crash?

  1. Redo: Replay committed transactions that might not have reached disk
  2. Undo: Roll back uncommitted transactions that were partially written
    Uses write-ahead logging (WAL)—log entries are written before data changes.

233. How are concurrent transactions managed?

Through isolation levels and locking:

  • READ UNCOMMITTED: No isolation, can see dirty data
  • READ COMMITTED: See only committed data
  • REPEATABLE READ: Same query returns same results
  • SERIALIZABLE: Full isolation, transactions appear sequential

234. Why are transactions important?

Real-world operations often involve multiple database changes that must happen together. Without transactions, partial completion causes data corruption and business errors.

235. What is rollback?

Undoing all changes made by a transaction, returning the database to its state before the transaction began. Happens on error or explicit ROLLBACK command.

236. What is commit?

Making all changes from a transaction permanent. After COMMIT, changes are durable and visible to other transactions.

SECTION 13 — SECURITY & ACCESS CONTROL

237. What is database security?

Protecting data from unauthorized access, modification, or destruction. Includes authentication, authorization, encryption, and auditing.

238. How do views improve security?

By restricting what users can see. Create views that exclude sensitive columns, and grant access to views instead of base tables.

239. What is row-level security?

Restricting access to specific rows based on user identity. A sales rep sees only their own customers’ rows.

240. What is column-level security?

Restricting access to specific columns. Hiding salary information from general employees while showing it to HR.

241. What is inference attack?

Deducing sensitive information from non-sensitive data. Example: Knowing department averages and all but one salary, you can calculate the last person’s salary.

242. How does DBMS restrict access?

  • Authentication: Verify identity (username/password, certificates)
  • Authorization: Grant specific privileges (SELECT, INSERT, UPDATE, DELETE)
  • Views: Limit visible data
  • Encryption: Protect data at rest and in transit

243. What is authorization?

Granting permissions to users—what they can do and what data they can access.

GRANT SELECT ON employees TO hr_manager;
REVOKE DELETE ON employees FROM intern;

244. What is authentication?

Verifying who someone is—checking credentials before allowing database access.

245. What is privilege management?

Controlling and tracking what operations each user can perform on which objects. Includes granting, revoking, and auditing privileges.

246. Why is abstraction important for security?

Because users interact with abstract views rather than raw physical storage. They can’t access what they can’t see, and they can’t manipulate what they don’t understand.

SECTION 14 — MODERN DATABASE CONCEPTS

247. What is ORM?

Object-Relational Mapping. A technique that lets you interact with databases using objects in your programming language, instead of writing raw SQL.

248. What is object-relational mapping?

The process of converting between relational database tables and object-oriented programming objects. Example: A User class maps to a users table.

# Django ORM example
user = User.objects.get(name='Alice')  # Instead of: SELECT * FROM users WHERE name='Alice'

249. What is impedance mismatch?

The fundamental differences between relational tables and programming objects:

  • Tables are flat; objects have nested structures
  • Tables use foreign keys; objects use references
  • SQL returns sets; programs expect individual objects
  • Object inheritance doesn’t map cleanly to tables

250. What is NoSQL?

“Not Only SQL”—database systems that don’t use the traditional relational model. Designed for scalability, flexibility, and specific data types.

Types:

  • Document (MongoDB): JSON-like flexible schemas
  • Key-Value (Redis): Simple hash table
  • Column (Cassandra): Wide columns, good for big data
  • Graph (Neo4j): Nodes and relationships

251. What is NewSQL?

Modern databases that combine NoSQL scalability with ACID transactions and SQL interface. Examples: Google Spanner, CockroachDB, TiDB.

252. What is cloud database?

Database services hosted and managed by cloud providers (AWS, Azure, GCP). You don’t manage hardware, backups, or patching—the provider does.

253. What is distributed database?

A database spread across multiple computers (nodes), often in different locations. Appears as a single database to users but stores data across a network.

254. How do modern databases extend ANSI/SPARC?

  • Adding federation layers (unifying multiple databases)
  • Caching layers between external and conceptual
  • Sharding at the internal level (splitting data across servers)
  • Replication views at external level (read replicas)

255. What is multi-tenancy?

Multiple customers (tenants) sharing one database instance while keeping data isolated. Each tenant sees only their own schema or data partition.

256. How do cloud databases manage abstraction?

Through managed services that hide infrastructure: auto-scaling, automated backups, serverless compute. Users see only the database connection; the cloud handles everything below.

257. What are examples of distributed SQL databases?

  • Google Spanner: Globally distributed, strongly consistent
  • CockroachDB: PostgreSQL-compatible, distributed
  • TiDB: MySQL-compatible, cloud-native
  • YugabyteDB: Distributed SQL with NoSQL flexibility

258. How do ORMs interact with relational models?

ORMs generate SQL automatically from object operations. They handle the mapping between class attributes and table columns, but sometimes generate inefficient queries.

SECTION 15 — REAL-WORLD IMPLEMENTATION

259. How does SQL implement ANSI/SPARC architecture?

  • External: Views (CREATE VIEW)
  • Conceptual: Base tables, constraints, relationships (CREATE TABLE)
  • Internal: Indexes, tablespaces, storage engines (CREATE INDEX, engine configuration)

260. How does DBMS translate SQL internally?

  1. Parser: Converts SQL text to parse tree
  2. Analyzer: Validates against catalog (table/column existence)
  3. Rewriter: Applies views and rules
  4. Optimizer: Generates execution plan
  5. Executor: Runs the plan against storage

261. How does logical schema map to physical storage?

The DBMS maintains a system catalog (data dictionary) that tracks:

  • Which files store which tables
  • Index definitions and their file locations
  • Column types and offsets within records
  • Partition boundaries

262. How do queries travel through DBMS layers?

User → External View → Conceptual Schema → Internal Schema → Storage Engine → Disk I/O → Results back up the chain

263. How are tables converted into bytes?

The DBMS uses record formats:

  • Fixed-length fields: Store directly with padding
  • Variable-length fields: Store length prefix + data
  • NULL bitmaps: Track which fields are NULL
  • Row headers: Store metadata (visibility info for MVCC)

264. How are rows stored physically?

In pages/slots within files. Each page has a header (free space info, checksum), then slot directory pointing to row locations, then actual row data.

265. How are indexes maintained internally?

  • B+ Trees: Balanced on insert/delete, nodes split/merge
  • Hash indexes: Rehash when load factor exceeds threshold
  • Bitmap indexes: Update bit vectors when rows change

All index changes are logged for recovery.

266. How does DBMS enforce constraints internally?

  • Primary/Unique: Check unique indexes before insert/update
  • Foreign Key: Check referenced table’s primary key index
  • CHECK: Evaluate expression for each row
  • NOT NULL: Reject NULLs during type checking

267. How do views hide schema complexity?

Views can join multiple tables, filter rows, compute columns, and rename fields. Users see a simple flat structure while the view definition handles the complexity.

-- User sees: customer_order_summary
-- Underlying: 4 tables joined with aggregation
CREATE VIEW customer_order_summary AS
SELECT c.name, COUNT(o.id) AS order_count, SUM(o.total) AS lifetime_value
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
LEFT JOIN order_items oi ON o.id = oi.order_id
LEFT JOIN products p ON oi.product_id = p.id
GROUP BY c.id, c.name;

268. How do modern DBMSs implement abstraction?

  • Storage engines: Pluggable backends (MySQL: InnoDB vs. MyISAM)
  • Query engines: Separate from storage (Apache Calcite)
  • Serverless layers: Abstract compute from storage (Snowflake, BigQuery)
  • API layers: GraphQL, REST over SQL

269. How does DBMS maintain independence between layers?

Through strict interfaces:

  • External layer only knows views and their definitions
  • Conceptual layer manages tables and constraints
  • Internal layer handles files and indexes
    Changes in one layer only require updating the mapping to adjacent layers.

270. How does abstraction help real-world scalability?

  • Vertical scaling: Upgrade hardware, internal layer adapts
  • Horizontal scaling: Add servers, internal layer shards/partitions
  • Read scaling: Add read replicas, external layer routes queries
  • Cloud scaling: Auto-scale storage, completely hidden from users

Applications keep running the same SQL while the DBMS handles massive infrastructure changes underneath.
