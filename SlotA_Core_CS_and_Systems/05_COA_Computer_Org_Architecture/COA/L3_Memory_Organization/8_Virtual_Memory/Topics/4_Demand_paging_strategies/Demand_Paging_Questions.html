1. Fundamental Understanding Questions

What is Demand Paging?
Demand Paging is a memory management technique where pages of a program are loaded into RAM only when they are needed (demanded) by the CPU, rather than loading the entire program at once. The OS creates the process structure, sets up a page table marking all pages as “Not Present,” and loads only the first page containing the entry point. Additional pages are loaded on-demand as page faults occur.

Why was Demand Paging invented?
It was invented to solve the fundamental mismatch between program sizes and available RAM. Early computers had limited RAM, and programs were growing larger. Loading everything was wasteful and slow. Demand Paging emerged as a way to run programs larger than physical memory while maintaining reasonable performance.

What problem does Demand Paging solve?
It solves three core problems:

  • RAM limitation: Programs larger than physical RAM can still run
  • Wasted resources: Unused code/data doesn’t occupy RAM
  • Slow startup: Programs begin executing immediately without waiting for full disk reads

Why can’t entire programs always fit into RAM?
Modern applications are massive (games: 50-100GB, databases: terabytes). RAM is expensive and physically limited. Even if RAM were large enough, loading unused portions wastes memory that could serve other processes or cache data.

How can a 10 GB program run on 8 GB RAM?
Through virtual memory and demand paging. Only the actively used pages (working set) reside in RAM at any moment. The rest stays on disk. The OS swaps pages in and out transparently. The program “feels” like it has 10GB because the virtual address space is that large, even though physical RAM is only 8GB.

What is the relationship between RAM and disk?
RAM is fast, expensive, volatile memory used for active data. Disk is slow, cheap, persistent storage used as backing store. In demand paging, disk extends RAM: together they form the virtual memory space. Disk holds pages not currently in RAM; RAM holds the working set.

Why is memory organized as a hierarchy?
Because of the speed-cost trade-off: faster memory is exponentially more expensive per byte. The hierarchy (registers → cache → RAM → disk) balances performance and cost by keeping frequently accessed data in fast memory and rarely accessed data in slow, cheap memory.

Why is fast memory expensive?
Fast memory (SRAM in caches, DRAM in RAM) requires more transistors, complex manufacturing, lower density, and higher power consumption. The physics of faster switching speeds demand premium materials and designs.

Why is slow memory cheap?
Slow memory (HDD, SSD) uses higher-density storage technologies. Hard drives use magnetic platters; SSDs use flash memory cells that are slower but denser and cheaper to produce at scale.

What is the illusion of infinite memory?
Virtual Memory creates the perception that the computer has unlimited RAM because programs can address a virtual address space much larger than physical RAM. The OS transparently manages the backing store (disk), making the program unaware that some “memory” is actually on disk.

What is Virtual Memory?
A memory management abstraction that gives each process its own private address space, decoupled from physical RAM. It allows programs to use more memory than physically available by using disk as an extension, and provides isolation between processes.

How does Virtual Memory relate to Demand Paging?
Virtual Memory is the concept; Demand Paging is the implementation mechanism. Virtual Memory defines the address space abstraction; Demand Paging is the technique that makes it practical by loading pages only when needed.

Why do modern operating systems require Demand Paging?
Without it, multitasking would collapse. Multiple large programs couldn’t coexist. Startup times would be minutes. RAM would be exhausted instantly. The entire modern computing experience—running browsers with hundreds of tabs, large games, IDEs—depends on it.

What would happen without Demand Paging?

  • Programs couldn’t exceed RAM size (hard cap)
  • Startup would require reading entire programs from disk (5-10 seconds for 500MB)
  • Multitasking would be severely limited (each program hogs full RAM)
  • RAM would be filled with unused code/data (~70% waste)
  • System would be unusable for modern workloads

Why is Demand Paging considered revolutionary?
It decoupled program size from RAM size for the first time, enabling:

  • Programs larger than physical memory
  • Fast startup (programs appear “instantly”)
  • Efficient multitasking (many programs share RAM via partial loading)
  • The foundation of modern operating systems

2. Memory Hierarchy Questions

What is the memory hierarchy?
A pyramid structure of memory technologies organized by speed, cost, and size:

Level Speed Cost Size Example
Registers ~1ns Highest ~1KB CPU registers
L1 Cache ~2-4ns Very High ~32-64KB On-chip SRAM
L2/L3 Cache ~10-20ns High ~1-32MB CPU cache
RAM ~100ns Moderate 8-128GB DRAM
SSD ~100μs Low 256GB-4TB Flash storage
HDD ~10ms Lowest 1-16TB Magnetic disk

Why do computers use multiple layers of memory?
To exploit locality of reference. Programs tend to reuse recently accessed data (temporal locality) and access nearby data (spatial locality). The hierarchy keeps hot data in fast, small memory and cold data in slow, large memory.

What are registers?
The fastest, smallest memory inside the CPU. They hold data the CPU is actively computing on. Access is nearly instantaneous (~1 clock cycle). There are typically 16-32 general-purpose registers in modern CPUs.

What is RAM?
Random Access Memory—volatile, byte-addressable memory where active program data and code reside. DRAM (Dynamic RAM) is used for main memory; it requires periodic refreshing and is slower than cache but much larger.

What is secondary storage?
Non-volatile storage (HDD, SSD) that persists data without power. It’s 100,000× slower than RAM but 100× cheaper per GB. It serves as the backing store for virtual memory.

Why is disk slower than RAM?

  • Physical movement: HDDs have mechanical heads that must move to the correct track (seek time: ~5-10ms) and wait for the platter to rotate (latency: ~4ms)
  • Electronic vs. mechanical: RAM is pure electronics; disks involve physics
  • Bandwidth: Even SSDs are limited by NAND flash read speeds (~500MB/s vs RAM’s ~25GB/s)
  • Latency: RAM access is ~100ns; SSD is ~100μs (1000× slower); HDD is ~10ms (100,000× slower)

Why are memory technologies organized as a pyramid?
The pyramid reflects the fundamental trade-off: you can have fast, small, expensive memory OR slow, large, cheap memory—not both. The hierarchy places the most critical data at the top (fastest) and everything else at the bottom.

What trade-off exists between memory speed and cost?
Speed ∝ 1/Cost per GB. Each 10× speed increase roughly costs 10× more per GB. This is why we can’t build computers with 1TB of register-speed memory—it would cost millions.

Why do we keep only active data in RAM?
Because RAM is expensive and limited. Keeping inactive data in RAM wastes space that could hold other active data, reducing the number of programs that can run simultaneously and hurting cache performance.

3. Swapping vs Demand Paging Questions

What is swapping?
An older memory management technique where the entire process is moved between RAM and disk. When RAM is full, an entire process is swapped out to disk; when needed again, the entire process is swapped back in.

How did old swapping systems work?
The OS would:

  1. Pause a process when RAM was full
  2. Write its entire memory image to a swap partition on disk
  3. Load another complete process from disk into RAM
  4. Resume execution

Why was swapping inefficient?

  • Massive I/O: Moving entire processes (hundreds of MB) to disk is slow
  • Wasted bandwidth: Even unused parts of the program were swapped
  • Poor granularity: No way to keep active parts in RAM while moving inactive parts out
  • Thrashing risk: Heavy swapping could paralyze the system

What is the difference between swapping and Demand Paging?

Aspect Swapping Demand Paging
Unit of transfer Entire process Individual pages (4KB)
Granularity Coarse Fine
RAM usage All-or-nothing Partial, precise
Startup Load everything first Load only first page
Flexibility Low High
Efficiency Poor Excellent

Why is loading entire programs wasteful?
Programs contain sections used at different times or never:

  • Error handling code (rarely used in normal operation)
  • Initialization code (used once at startup)
  • Features for edge cases (unused 99% of the time)
  • Debug/info sections (never in production)

A typical large program uses only ~15% of its code/data in any 10-second window.

Why is Demand Paging better than swapping?

  • Fine granularity: Only needed pages are loaded
  • Better RAM utilization: Multiple programs share RAM efficiently
  • Faster startup: Programs begin executing in milliseconds
  • Less I/O: Only active pages touch the disk

Why is startup faster with Demand Paging?
The OS loads only the first page (~4KB) containing the entry point. The program begins executing immediately. The rest loads lazily over time. With swapping/loading-everything, the entire program must be read from disk first.

Why does swapping hurt multitasking?
Each process requires its full memory footprint in RAM. With swapping, only a few large processes can fit. Demand paging allows dozens of processes to share RAM by each using only their active pages.

Why does Demand Paging improve multitasking?
RAM becomes a shared pool of frames allocated dynamically to the pages actually being used. Processes don’t “own” RAM; they borrow it page-by-page. This allows many more processes to run concurrently.

4. Page Fundamentals Questions

What is a page?
A fixed-size block of virtual memory (typically 4KB), the basic unit of transfer between disk and RAM. It’s the atomic unit of memory management—the smallest amount that can be independently mapped, loaded, or evicted.

What is a frame?
A page-sized slot in physical RAM. If pages are 4KB, frames are 4KB. A frame holds exactly one page when that page is resident in RAM.

What is the difference between a page and a frame?

  • Page: Virtual concept—belongs to a process’s virtual address space
  • Frame: Physical concept—a location in actual RAM hardware
    A page is “mapped to” a frame when loaded in RAM.

Why are pages fixed-size?
Fixed size simplifies hardware and software:

  • Hardware: MMU can use simple bit-shifting to split addresses
  • OS: Easy frame allocation (any free frame fits any page)
  • Disk I/O: Predictable transfer sizes
  • Replacement: Uniform eviction decisions

Why is page size usually 4 KB?
4KB balances several factors:

  • Small enough: Minimizes internal fragmentation (wasted space in last page)
  • Large enough: Amortizes page table overhead and disk seek costs
  • Hardware tradition: x86 architecture standardized on 4KB; compatibility matters
  • Locality: Most programs exhibit locality within 4KB regions

Why are powers of two used in memory systems?
Powers of two make address translation trivial via bit manipulation:

  • Page number = high bits of address
  • Offset = low bits of address
    No division or multiplication needed—just bit masking and shifting, which hardware performs in nanoseconds.

What is internal fragmentation?
Wasted space within the last page of a segment. If a program needs 4097 bytes, it gets 2 pages (8192 bytes), wasting 4095 bytes in the second page. With 4KB pages, average internal fragmentation is 2KB per segment.

Why is fixed-size paging easier for hardware?
The MMU can use simple arithmetic:

Page Number = Address >> 12    (divide by 4096)
Offset = Address & 0xFFF       (modulo 4096)

Variable sizes would require complex division, lookup tables, or software assistance.

How are addresses divided into page number and offset?
For 32-bit addresses with 4KB (2¹²) pages:

  • Page Number: Upper 20 bits (bits 12-31) → indexes into page table
  • Offset: Lower 12 bits (bits 0-11) → added to frame base address

Example: 0x00005078

  • Binary: 0000 0000 0000 0000 0101 | 0000 0111 1000
  • Page Number: 0x5 (upper 20 bits)
  • Offset: 0x078 (lower 12 bits)

How does page alignment work?
Pages must start at addresses divisible by the page size. A 4KB page starting at 0x0000 spans 0x0000-0x0FFF. The next page starts at 0x1000. This alignment ensures the offset fits in 12 bits and pages don’t overlap.

5. Address Translation Questions

What is a virtual address?
An address generated by the CPU that belongs to a process’s private address space. It has no direct physical meaning—the MMU must translate it to a physical address before memory access.

What is a physical address?
The actual address placed on the memory bus to access RAM hardware. It corresponds to a specific DRAM chip, bank, row, and column.

How does address translation work?

  1. CPU generates virtual address
  2. MMU splits it into page number + offset
  3. MMU looks up page number in page table
  4. Page table yields frame number (if page is present)
  5. MMU concatenates frame number + offset → physical address
  6. Physical address sent to RAM

What is the MMU?
Memory Management Unit—a hardware component (often integrated into the CPU) that performs virtual-to-physical address translation, enforces memory protection, and detects page faults.

What does the MMU do?

  • Translates every memory access from virtual to physical
  • Checks protection bits (read/write/execute)
  • Sets page fault flag when translation fails
  • Manages the TLB (translation cache)
  • Updates reference/dirty bits

How does the MMU translate addresses?

Physical Address = (Frame_Number × Page_Size) + Offset

Or equivalently: shift frame number left by page size bits, then add offset.

How does the MMU detect missing pages?
The MMU checks the Valid/Present bit in the page table entry:

  • Valid = 1: Page is in RAM → proceed with translation
  • Valid = 0: Page not in RAM → trigger page fault exception

What is the role of the page table?
The page table is the data structure that maps virtual page numbers to physical frame numbers (or disk locations). It’s the “directory” that makes virtual memory possible. Each process has its own page table.

What is stored inside a Page Table Entry (PTE)?

Field Bits Purpose
Frame Number 20 Physical frame address
Valid/Present 1 Is page in RAM?
Dirty/Modified 1 Has page been written?
Reference/Accessed 1 Has page been read/written?
Protection (R/W/X) 3 Read/Write/Execute permissions
Cacheable 1 Can be cached?
Global 1 Shared across processes?

What is the Valid Bit?
Indicates whether the page is currently resident in physical RAM (1 = present, 0 = not present). If 0, accessing this page triggers a page fault.

What is the Dirty Bit?
Set to 1 when the page is modified (written to). Tells the OS this page must be written back to disk before eviction (otherwise changes are lost).

What is the Reference Bit?
Set to 1 when the page is accessed (read or written). Used by replacement algorithms (like Clock) to identify recently used pages. The OS periodically clears this bit to track usage over time.

What are protection bits?
Flags controlling access permissions:

  • Read ®: Can the page be read?
  • Write (W): Can the page be modified?
  • Execute (X): Can code in this page be executed?
    Violations trigger protection faults (e.g., segmentation fault).

What happens during a page table lookup?

  1. MMU extracts page number from virtual address
  2. MMU reads PTE from page table in RAM
  3. MMU checks Valid bit
  4. If valid: extracts frame number, computes physical address
  5. If invalid: signals page fault to CPU

How does the MMU compute physical addresses?

Physical Address = (Frame_Number << 12) | Offset

For 4KB pages: shift frame number left by 12 bits (multiply by 4096), then bitwise-OR with the 12-bit offset.

6. Page Fault Questions

What is a page fault?
A hardware exception triggered when the CPU accesses a virtual page not currently resident in physical RAM. It’s the mechanism that enables demand paging—the “signal” that a needed page must be loaded from disk.

Is a page fault an error?
No—it’s a normal, expected event in demand paging systems. Unlike segmentation faults (illegal access), page faults are part of correct operation. The OS handles them transparently.

What triggers a page fault?
Any memory access (read, write, or execute) to a page where the page table’s Valid bit is 0 (page not in RAM).

What happens when a page fault occurs?
The complete sequence:

  1. CPU generates virtual address
  2. MMU checks page table → Valid = 0
  3. MMU raises page fault exception
  4. CPU switches to kernel mode
  5. CPU saves state (registers, PC, status) to stack
  6. OS page fault handler runs
  7. OS validates access (legal? permissions?)
  8. OS locates page on disk (or allocates zero page)
  9. OS finds free frame (or evicts victim page)
  10. OS initiates disk read (DMA transfer to frame)
  11. CPU context-switches to another process (optional)
  12. Disk interrupt signals completion
  13. OS updates PTE: Valid=1, Frame=assigned
  14. OS restores CPU state (registers, PC)
  15. CPU returns to user mode, re-executes faulting instruction
  16. Instruction succeeds (page now in RAM)

What are the steps in page fault handling?
See above—16 steps from fault detection to instruction restart.

What does the OS do during a page fault?
The OS acts as the “librarian”:

  • Validates the access is legal (not a segfault)
  • Finds where the page lives on disk
  • Allocates a physical frame
  • Initiates disk I/O (often via DMA)
  • Updates page tables when I/O completes
  • Restarts the faulting process

Why does the CPU switch to kernel mode?
Page fault handling requires privileged operations: accessing page tables, initiating disk I/O, modifying process state, and restarting instructions. These cannot be done in user mode for security and stability.

What is a trap?
A synchronous exception caused by the currently executing instruction (like page fault, division by zero, system call). The CPU automatically transfers control to a kernel handler.

What is a hardware exception?
An event detected by CPU hardware that requires OS intervention. Page faults are hardware exceptions because the MMU (hardware) detects the missing page and signals the CPU.

What state does the CPU save during a fault?

  • Program Counter (PC): address of faulting instruction
  • Processor Status Register (PSR): flags, mode bits
  • General-purpose registers (pushed to kernel stack)
  • Stack pointer

Why is the instruction restarted after a fault?
The faulting instruction never completed—it was interrupted mid-execution. After the page is loaded, the instruction must re-execute to access the now-present page and complete its operation.

Why does the saved PC point to the faulting instruction?
So that when the OS returns via IRET, execution resumes at the exact instruction that caused the fault—not the next one. This allows the instruction to succeed on the second attempt.

How does execution continue after paging?
After IRET, the CPU is back in user mode with the PC pointing to the faulting instruction. The MMU now finds Valid=1 in the page table, translates the address successfully, and the instruction completes normally.

What is the difference between a major and minor page fault?

  • Major page fault: Page must be read from disk (slow: ~10ms)
  • Minor page fault: Page is already in RAM (e.g., shared library already loaded by another process, or zero page allocation). Much faster: no disk I/O needed.

What is an invalid page fault?
A page fault for an address that is not part of the process’s legal address space (e.g., dereferencing a null pointer, accessing unmapped memory). The OS cannot satisfy this—it’s a programming error.

What causes a segmentation fault?
An invalid memory access: writing to read-only memory, executing non-executable memory, or accessing addresses outside allocated regions. The OS kills the process with SIGSEGV.

How does the OS validate memory access?
The OS checks:

  1. Is the virtual address within a valid VMA (Virtual Memory Area)?
  2. Are the access permissions (R/W/X) sufficient?
  3. Is the page mapped to a file or swap space?
  4. Is it a demand-zero page?

What is a legal vs illegal memory access?

  • Legal: Address belongs to process, permissions match operation, page exists in backing store
  • Illegal: Address outside process space, permission violation (write to read-only), unmapped address

7. Page Replacement Questions

What happens when RAM becomes full?
When all frames are occupied and a new page is needed, the OS must evict (remove) an existing page from RAM to make space.

What is page replacement?
The algorithm and process of selecting which resident page to remove from RAM when a new page needs to be loaded.

Why must pages be evicted?
RAM is finite. To load a new page, an old one must go. The goal is to evict the page least likely to be needed soon.

What is a victim page?
The page selected by the replacement algorithm to be removed from RAM.

How does the OS choose a victim page?
Using replacement algorithms that predict future usage based on past behavior.

What is FIFO replacement?
First-In-First-Out: evict the page that has been in RAM the longest. Simple but ignores usage patterns. Can suffer from Belady’s Anomaly: more frames can sometimes increase faults.

What is LRU replacement?
Least Recently Used: evict the page that hasn’t been accessed for the longest time. Assumes temporal locality—if a page hasn’t been used recently, it won’t be used soon. Excellent in theory but expensive to implement perfectly.

What is the Optimal algorithm?
OPT or MIN: evict the page that won’t be used for the longest time in the future. This is the theoretical best but impossible to implement because it requires knowledge of the future. Used as a benchmark.

What is the Clock algorithm?
An efficient approximation of LRU using a circular buffer and a reference bit:

  • Pages arranged in a circle (like a clock)
  • Each page has a reference bit (set on access)
  • When seeking a victim, the hand sweeps around
  • If reference bit = 1: clear it, move on
  • If reference bit = 0: this is the victim
    Combines reasonable performance with O(1) overhead.

Why is no replacement algorithm perfect?
All algorithms must predict the future. Programs have complex, non-deterministic access patterns. What works well for one workload fails for another. The Optimal algorithm is unachievable; all real algorithms are approximations.

What trade-offs exist in replacement algorithms?

  • Accuracy vs. overhead: Better prediction requires more bookkeeping
  • Fairness vs. performance: Should all processes be treated equally?
  • Global vs. local: Should replacement consider all pages or per-process?

Why is LRU better than FIFO?
LRU exploits temporal locality—recently used pages are likely to be used again. FIFO ignores usage entirely; a frequently used page can be evicted just because it loaded early.

Why is Optimal impossible in practice?
It requires perfect knowledge of all future memory accesses. The OS cannot predict what the program will do next. It’s a theoretical lower bound used to evaluate real algorithms.

8. Dirty/Clean Page Questions

What is a clean page?
A page that has not been modified since loading from disk. Its contents match the disk copy exactly. Can be discarded without writing back.

What is a dirty page?
A page that has been written to (modified). Its RAM contents differ from the disk copy. Must be written back to disk before eviction to preserve changes.

Why must dirty pages be written back to disk?
To prevent data loss. If a dirty page is discarded without saving, the modifications (user data, program state) are permanently lost.

Why can clean pages be discarded immediately?
The disk already has an identical copy. No write is needed—just remove the page from RAM and reuse the frame.

How does the system know a page is dirty?
The MMU sets the Dirty Bit in the PTE when a write operation occurs to that page. The OS checks this bit during replacement.

What happens if dirty pages are not saved?
Data corruption and loss. File modifications disappear. Program state is corrupted. Database transactions may be lost. System crashes become catastrophic.

Why are dirty pages expensive to evict?
They require a disk write before the frame can be reused. Disk writes are slow (~10ms for HDD, ~100μs for SSD). Clean pages are instant—just discard.

Why do systems try to minimize dirty pages?
To make eviction faster. Techniques include:

  • Write-through caching: Write to disk immediately
  • Background writeback: Periodically flush dirty pages
  • Copy-on-Write: Delay writes until necessary

9. Working Set & Locality Questions

What is a working set?
The set of pages a process is actively using during a specific time window. It’s the “footprint” of the program’s current activity. If the working set fits in RAM, page faults are minimal.

Why do programs only use part of themselves at once?
Programs have phases: initialization, main loop, error handling, cleanup. Different phases need different code/data. A game doesn’t need its settings menu code during gameplay.

What is locality of reference?
The tendency of programs to access a relatively small portion of their address space frequently, rather than spreading accesses uniformly across all memory.

What is temporal locality?
If a memory location is accessed, it’s likely to be accessed again soon. Example: loop variables, frequently called functions.

What is spatial locality?
If a memory location is accessed, nearby locations are likely to be accessed soon. Example: sequential array traversal, loading a texture and then using adjacent pixels.

Why does Demand Paging depend on locality?
Because locality ensures that once a page is loaded, multiple accesses will hit in RAM before another fault occurs. Without locality, every access would fault, and the system would collapse.

What happens if locality does not exist?
Every access would likely be to a different page, causing a page fault each time. The system would spend 100% of its time swapping—this is thrashing. Effective Access Time would equal disk access time (~10ms), making the computer unusable.

Why do sequential accesses perform well?
They exhibit strong spatial locality. Accessing array[0] loads a page containing array[0] through array[1023] (for 4-byte ints). The next 1023 accesses hit in RAM.

Why do random accesses cause problems?
They defeat both temporal and spatial locality. Each access may target a different page, causing constant page faults. The working set becomes enormous.

Why does array traversal work efficiently?
Arrays are contiguous in memory. Traversing sequentially stays within the same page for many iterations. One page fault serves hundreds or thousands of accesses.

Why does loading one page help many accesses?
A 4KB page holds 1024 integers (4 bytes each). Loading the page for scores[0] also brings in scores[1] through scores[1023]. The next 1023 accesses are hits.

10. Thrashing Questions

What is thrashing?
A state where the system spends most of its time handling page faults rather than executing useful instructions. The CPU is constantly waiting for disk I/O.

Why does thrashing occur?
When the total working set of all running processes exceeds available RAM. The OS desperately tries to keep active pages in RAM but must constantly evict and reload pages.

What are the symptoms of thrashing?

  • System becomes extremely slow (seconds to respond)
  • Disk activity at 100% (constant grinding on HDDs)
  • CPU utilization drops (CPU waiting, not computing)
  • Programs freeze or stutter
  • Mouse/keyboard input lags severely

Why does the system freeze during thrashing?
Every memory access triggers a page fault. The CPU switches to kernel mode, initiates disk I/O, waits milliseconds, then resumes—only to fault again immediately. No forward progress is made.

Why can’t replacement algorithms fix thrashing?
The problem isn’t which page to evict—it’s that there simply isn’t enough RAM for the working sets. No algorithm can fit 16GB of working set into 8GB of RAM.

What causes excessive page faults?

  • Too many processes running simultaneously
  • Processes with large working sets
  • Random memory access patterns (no locality)
  • Insufficient RAM for the workload

What happens when working sets exceed RAM?
The OS enters a cycle: load page A (evict B), then fault on B (evict A), then fault on A again… Pages are constantly swapped in and out with no productive work.

Why does disk usage become 100% during thrashing?
The disk is constantly reading pages in and writing dirty pages out. The queue of I/O requests grows, and the disk head thrashes (on HDDs) or NAND channels saturate (on SSDs).

How can thrashing be solved?

  • Add more RAM (hardware solution)
  • Kill or suspend processes (reduce total working set)
  • Working set model: Monitor each process’s working set and only admit processes whose working sets fit in available RAM
  • Page fault frequency (PFF) control: If fault rate too high, suspend a process

11. Performance Questions

Why are page faults expensive?
A page fault requires:

  • CPU mode switch (user → kernel)
  • OS handler execution
  • Disk I/O initiation (DMA setup)
  • Disk read (~10ms HDD, ~100μs SSD)
  • Disk interrupt handling
  • Page table updates
  • Mode switch back (kernel → user)
  • Instruction restart

Why is disk access slow?
HDDs are mechanical: seek time (~5-10ms) + rotational latency (~4ms) + transfer time. Even SSDs, while electronic, have NAND flash latency and controller overhead.

Why are HDDs slower than SSDs?
HDDs have moving parts (spinning platters, moving heads). SSDs are solid-state with no mechanical delays. SSD access: ~100μs; HDD access: ~10ms (100× slower).

Why are SSDs still slower than RAM?

  • Interface: SSDs connect via SATA/NVMe; RAM connects directly to the memory bus
  • NAND latency: Flash memory cells take microseconds to read; DRAM takes nanoseconds
  • Controller overhead: SSDs have complex controllers for wear leveling, error correction
  • Bandwidth: RAM bandwidth is ~25-50 GB/s; SSDs are ~0.5-7 GB/s

Why is RAM much faster than disk?
RAM is directly connected to the CPU via high-speed memory buses. It’s purely electronic with nanosecond-scale access. No mechanical movement, no controller translation, no protocol overhead.

What happens to CPU performance during page faults?
The CPU stalls. It switches to the OS handler, then often idles waiting for disk I/O. During that time, no useful computation occurs on the faulting process. The CPU may context-switch to another process, but that process may also fault.

Why does one page fault dominate performance?
Disk access (~10ms) is 100,000× slower than RAM access (~100ns). A single page fault costs as much as 100,000 RAM accesses. Even a 0.1% fault rate makes disk time dominate total execution time.

Why can tiny fault rates cause huge slowdowns?
Because of the extreme speed differential. At 0.1% fault rate with 10ms disk:

EAT = 0.999 × 100ns + 0.001 × 10,000,000ns = 100ns + 10,000ns = 10,100ns

That’s a 100× slowdown from a 0.1% fault rate!

Why does the CPU become idle during faults?
After initiating disk I/O (which is handled by DMA), the CPU has nothing to do for the faulting process until the disk interrupt arrives. The OS may schedule another process, but if all processes are waiting for disk, the CPU sits idle.

Why are memory systems sensitive to page faults?
The 100,000× speed gap between RAM and disk means page faults are catastrophic outliers. A system with 99.9% hit rate can still be 100× slower than a system with 100% hit rate.

13. Hardware-Level Questions

What hardware components participate in paging?

  • CPU: Generates virtual addresses, handles exceptions
  • MMU: Translates addresses, checks protection, detects faults
  • TLB: Caches recent translations
  • RAM: Holds page tables and resident pages
  • Disk Controller/DMA: Handles disk I/O without CPU involvement
  • Interrupt Controller: Signals disk completion

What is the role of the CPU?
Generates virtual addresses during instruction execution. On fault, saves state and transfers to kernel mode. After handling, restarts the faulting instruction.

What is the role of the MMU?
The central translator. Every memory access goes through the MMU. It performs address translation, protection checks, and fault detection in hardware.

What is the role of the OS kernel?
The software handler for page faults. It manages page tables, decides which pages to load/evict, initiates disk I/O, and updates mappings. The kernel is the “policy” layer; hardware is the “mechanism” layer.

What is the Interrupt Vector Table?
A data structure (set by the OS at boot) that maps exception types (including page fault) to handler addresses. When a page fault occurs, the CPU looks up the handler address in this table and jumps to it.

What is DMA (Direct Memory Access)?
A hardware mechanism that allows devices (like disk controllers) to transfer data directly to/from RAM without CPU involvement. The CPU sets up the transfer, then the DMA controller handles it independently.

How does disk DMA work?

  1. OS tells DMA controller: “Read page X from disk to frame Y in RAM”
  2. DMA controller coordinates disk → RAM transfer
  3. CPU is free to do other work
  4. DMA controller interrupts CPU when done
  5. OS updates page table and resumes process

What happens during a hardware trap?

  1. CPU completes current instruction (or aborts it)
  2. CPU switches to kernel mode
  3. CPU saves PC, PSR, registers to kernel stack
  4. CPU loads new PC from Interrupt Vector Table (handler address)
  5. Execution continues in kernel handler

How does the CPU enter kernel mode?
Automatically on exceptions (like page faults). The CPU sets the mode bit in the PSR, switches to the kernel stack, and jumps to the handler. This is a hardware mechanism—programs cannot fake it.

How does the CPU return to user mode?
Via the IRET (Interrupt Return) instruction. This restores the saved PC, PSR (including user mode bit), and registers from the kernel stack, resuming the interrupted user program.

What is the TLB?
Translation Lookaside Buffer—a small, fast hardware cache of recent virtual-to-physical translations. It sits between the CPU and page table to speed up address translation.

Why is the TLB necessary?
Without TLB, every memory access would require two RAM accesses: one to read the page table, one to access the data. The TLB caches translations, reducing this to one RAM access (on TLB hit).

What is a TLB hit?
The virtual page number is found in the TLB. The MMU immediately has the frame number—no page table lookup needed. Translation takes nanoseconds.

What is a TLB miss?
The virtual page number is not in the TLB. The MMU must read the page table from RAM to find the translation, then cache it in the TLB for future use. Slower, but still no page fault if the page is resident.

Why would memory access require two RAM accesses without TLB?
First access: read Page Table Entry from RAM to get frame number. Second access: use frame number to read/write actual data. The TLB eliminates the first access for cached translations.

Why must the TLB be flushed on context switches?
Each process has its own page table and virtual address space. The same virtual number maps to different physical frames in different processes. If the TLB isn’t flushed, Process B could access Process A’s physical memory—a security disaster.

14. Operating System Questions

How does the OS manage pages?
Through page tables, frame allocation tracking, replacement algorithms, and disk backing store management. The OS maintains data structures for all pages across all processes.

How does the OS track free frames?
Using a free frame list or bitmap. When a page needs loading, the OS takes a frame from this list. When a page is evicted, its frame is returned to the list.

How does the OS locate pages on disk?
Each PTE (when Valid=0) stores a disk address or pointer to a swap slot. The OS maintains swap space maps and file offset tables for memory-mapped files.

What is backing store?
The disk area where non-resident pages are kept. It includes swap space and memory-mapped files. It “backs” the virtual memory—providing the actual storage for pages not in RAM.

What is swap space?
A dedicated disk partition or file used exclusively for paging. When RAM is full, clean pages are discarded and dirty pages are written to swap space. It’s slower than RAM but essential for virtual memory.

What is a memory-mapped file?
A file on disk that is mapped directly into a process’s virtual address space. Accessing the memory reads/writes the file transparently. Used for efficient file I/O and shared libraries.

What is demand-zero paging?
When a program requests new memory (e.g., malloc, stack expansion) but hasn’t written to it yet, the OS maps it to a special “zero page” (a read-only page filled with zeros). Only when the program writes does Copy-on-Write create a real writable page.

What is Copy-on-Write (COW)?
An optimization where parent and child processes share physical pages after fork(). Pages are marked read-only. When either process writes, a private copy is created just for that process. Avoids unnecessary copying of unchanged pages.

How does the OS validate addresses?
By checking Virtual Memory Areas (VMAs)—data structures describing valid address ranges for each process. An address is valid only if it falls within a VMA with appropriate permissions.

What are VMAs (Virtual Memory Areas)?
Kernel data structures describing contiguous regions of a process’s virtual address space. Each VMA has start/end addresses, permissions (R/W/X), and backing file information. Page faults check against VMAs to validate accesses.

How does the kernel kill illegal processes?
On invalid access (segfault), the kernel sends SIGSEGV (signal 11) to the process. The default handler terminates the process and may generate a core dump. The kernel logs the event for debugging.

15. Startup & Efficiency Questions

Why does Demand Paging improve startup speed?
Only the first page (~4KB) containing the entry point is loaded. The program begins executing in milliseconds. The rest loads on-demand over time. Without it, a 500MB program would require 5+ seconds of disk reading before any execution.

Why do applications appear instantly?
The window appears after the first few pages load (the UI framework, initial code). The OS creates the process, maps the executable, loads the entry point, and starts execution. The user sees responsiveness while 99% of the program remains on disk.

What pages are initially loaded?
Typically just the page containing the program’s entry point (first instruction). Sometimes the OS also prefetches a few adjacent pages containing initialization code.

Why are most program pages unused initially?
Programs have “cold” regions: settings dialogs, error handlers, help systems, plugins, background services. These aren’t needed for basic operation. They load only if the user accesses them.

How does lazy loading work?
Pages are loaded only when accessed. The first access triggers a fault; subsequent accesses hit in RAM. This spreads disk I/O over time, avoiding upfront costs.

What is prefetching?
The OS or hardware predicts which pages will be needed soon based on access patterns and loads them in the background before they’re requested. Example: after loading page 5, prefetch pages 6 and 7.

How does prefetching improve performance?
It converts demand faults into hits. If the prediction is correct, the page is already in RAM when needed—no stall. Modern CPUs have hardware prefetchers that detect sequential and strided access patterns.

Why is loading everything inefficient?

  • Time: Reading 100MB from disk takes seconds
  • RAM waste: Unused pages occupy frames that could serve other processes
  • I/O waste: Disk bandwidth consumed by data never accessed
  • Energy: Unnecessary disk reads drain battery

How does Demand Paging optimize RAM usage?
Only active pages consume RAM. A 100MB program using 7MB of active code/data occupies only 7MB of RAM—93% savings. This allows more processes to coexist and more disk cache.

What are hot, warm, and cold pages/data?

  • Hot: Frequently accessed, must stay in RAM (working set)
  • Warm: Occasionally accessed, can tolerate rare faults
  • Cold: Rarely or never accessed, can stay on disk indefinitely

16. Real System Questions

How does Linux use Demand Paging?
Linux implements demand paging through:

  • Page fault handler: do_page_fault() in arch/x86/mm/fault.c
  • VMA management: mm_struct with vm_area_struct list
  • Swap subsystem: Swap partitions/files with swap_info_struct
  • OOM killer: Kills processes when RAM is exhausted
  • zswap: Compresses pages before swapping

How does Windows implement Demand Paging?
Windows uses:

  • Working Set Model: Tracks and limits each process’s working set
  • Standby List: Pages removed from working sets but kept in RAM for quick recovery
  • Modified Page Writer: Background thread flushing dirty pages
  • SuperFetch: Prefetches likely-needed pages based on usage history
  • ReadyBoost: Uses USB flash as cache

How do Android and iOS use paging?
Mobile OSes use demand paging with constraints:

  • No swap on early Android/iOS: Flash storage wears out; instead use LMK (Low Memory Killer)
  • Memory pressure: Kill background apps when RAM is low
  • Compressed RAM: zRAM (Android) stores compressed pages in RAM instead of disk
  • Frozen apps: iOS “freezes” background apps, keeping their pages but not scheduling them

What is swap compression?
Techniques like zswap and zram that compress pages in RAM before writing to disk (or instead of disk). Compression is faster than disk I/O, and compressed pages take less RAM.

What are huge pages?
Pages larger than standard 4KB—typically 2MB or 1GB. They reduce TLB misses for large datasets (one TLB entry covers 2MB instead of 512× 4KB entries). Used for databases, scientific computing, and virtualization.

Why are huge pages useful?

  • Fewer TLB misses: Large programs need fewer TLB entries
  • Faster page table walks: Shallower hierarchies
  • Better cache utilization: Less page table memory
  • Efficient for large buffers: Databases, GPU memory mappings

What modern optimizations exist for paging?

  • zswap/zram: Compressed swap in RAM
  • Huge pages: 2MB/1GB pages for large workloads
  • NUMA-aware allocation: Allocate pages near the accessing CPU
  • Transparent huge pages: Automatic promotion of contiguous 4KB pages to 2MB
  • Prefetching: Hardware and software predictive loading
  • Swap on SSD: Faster than HDD but still slower than RAM

17. Deep Conceptual Questions

How does hardware cooperate with the OS?
Hardware (MMU, CPU, DMA, disk) provides the mechanism: address translation, fault detection, data transfer. The OS provides the policy: which pages to load, which to evict, how to schedule I/O. They cooperate through defined interfaces (exceptions, interrupts, system calls).

Why is paging transparent to programs?
Programs use virtual addresses and never see physical frames. The OS and hardware handle all translation and faulting. A program can’t tell if its memory is in RAM or disk—access time is the only hint.

Why does the program not know paging occurred?
The OS restarts the faulting instruction after loading the page. From the program’s perspective, the instruction simply took longer. No registers are modified (except by the instruction itself). The program state is preserved.

Why does Demand Paging create the illusion of infinite fast memory?
Programs can allocate and use virtual addresses far exceeding RAM. The OS makes this work by using disk as backing store. Programs “feel” like they have vast memory, though access to disk-backed pages is slow.

Why is Demand Paging fundamentally probabilistic?
It bets on locality—that future accesses will likely hit pages already loaded or nearby to recently loaded pages. There’s no guarantee; it’s a statistical optimization based on observed program behavior.

Why does modern computing depend on predictable memory behavior?
CPUs pipeline instructions, prefetch data, and cache aggressively. All these optimizations assume memory access patterns are predictable (local). Random access destroys performance at every level.

Why is locality one of the most important principles in computing?
It enables:

  • Caching (CPU caches, TLB, disk cache)
  • Prefetching (hardware and software)
  • Demand paging (efficient virtual memory)
  • Database indexing (B-trees vs hash tables)
  • Compiler optimization (loop transformations)
    Without locality, every optimization collapses.

19. Research-Level / Systems Thinking Questions

Why does modern multitasking depend on Demand Paging?
Without it, each program would need its full size in RAM. Running 20 Chrome tabs (each 100MB) would require 2GB just for Chrome, plus OS, plus other apps. Only demand paging allows 50+ processes to share 8-16GB RAM by each using only active pages.

Why is Demand Paging a cooperation between hardware and software?
Hardware (MMU) provides fast translation and fault detection—this must be in hardware for speed. Software (OS) provides intelligent decisions about which pages to load/evict—this requires complex policy that changes and must be flexible. Neither alone suffices.

What physical limitations make paging necessary?

  • RAM cost: Fast DRAM is ~$5/GB; disk is ~$0.02/GB
  • RAM density: Physical slots and power limit RAM capacity
  • Speed gap: No single technology provides both speed and capacity
  • Power: Fast memory consumes more power (critical for mobile)

Why do modern systems rely on statistical program behavior?
Because exact prediction is impossible. Systems use probability: “most programs exhibit locality,” “most pages in working set stay in working set,” “sequential access likely continues.” These aren’t guarantees but strong trends that optimize average case.

Why are page faults catastrophic for performance?
The 100,000× speed gap means one fault equals 100,000 RAM accesses in time. A 1% fault rate makes the system 1000× slower. Faults are “outliers” that dominate average performance despite being rare.

Why is memory management fundamentally about prediction and locality?
The OS must predict which pages will be needed (to prefetch) and which won’t (to evict). Locality makes prediction possible. Without locality, prediction is random guessing, and paging fails. The entire edifice of virtual memory rests on the empirical fact that programs exhibit locality.
